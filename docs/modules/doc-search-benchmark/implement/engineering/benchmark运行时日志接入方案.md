# benchmark 运行时日志接入方案

## 1. 文档目的

本文给出 `doc_search benchmark` 运行时主日志的工程接入方案，作为代码实施前的冻结依据。

## 2. 目标

本次接入目标只有三项：

1. 为 benchmark 增加单文件中文主时间线日志
2. 覆盖请求、响应、用户模拟、judge、收口全链路
3. 让 `摘要` 在运行时由程序自动生成

## 3. 非目标

- 不修改任何前后端业务代码
- 不引入新的外部日志依赖
- 不让 LLM 负责现场总结日志
- 不把 `runtime.log` 变成正式评分来源

## 4. 方案概览

### 4.1 文件方案

- 单次运行目录：
  - `benchmark/reports/runs/<run_id>/`
- 实时主日志：
  - `benchmark/reports/runs/<run_id>/runtime.log`
- 正式报告：
  - `benchmark/reports/runs/<run_id>/report.actual.json`
  - `benchmark/reports/runs/<run_id>/report.score.json`
- 原始证据：
  - `benchmark/reports/runs/<run_id>/raw/*.json`

### 4.2 格式方案

固定主行：

```text
时间 | 级别 | 事件 | 定位 | 结果 | 摘要
```

可选附加行：

```text
  详情: ...
  路径: ...
```

### 4.3 摘要方案

- 使用程序模板
- 直接消费结构化字段
- 不依赖 LLM

## 5. 接入顺序

### 5.1 第一步：日志器落地

新增独立日志写入器模块，负责：

- 统一格式化
- 自动摘要
- 实时 flush
- 维护单次运行目录

### 5.2 第二步：run / suite / case / attempt 外层接入

先接最外层事件，确保 benchmark 一启动就能看到进度：

- `运行开始`
- `套件开始`
- `用例开始`
- `尝试开始`
- `运行异常`

### 5.3 第三步：多轮运行器主链路接入

再接运行核心事件：

- `请求预处理`
- `发送请求`
- `收到响应`
- `识别澄清问题`
- `用户选择已提交`
- `发现能力缺口`
- `尝试停止`
- `尝试完成`

### 5.4 第四步：用户模拟 agent 内部链路接入

补齐 AI 用户模拟内部事件：

- `开始用户模拟决策`
- `用户模拟模型调用`
- `用户模拟输出非法`
- `用户模拟校验失败`
- `用户模拟符号决策`
- `完成用户模拟决策`
- `用户模拟决策失败`

### 5.5 第五步：judge 与报告接入

最后补：

- `合同判定完成`
- `文件判定完成`
- `页码判定完成`
- `轨迹分析完成`
- `报告写入完成`
- `运行完成`

## 6. 风险与控制

### 6.1 风险：日志行过长

控制方式：

- 主行固定 6 段
- 长文本放到 `详情:`
- 原始路径放到 `路径:`

### 6.2 风险：泄露内部 payload

控制方式：

- 主日志只保留摘要
- 原始 `selection_payload` 不进入主日志
- 原始证据继续只留在 `raw/*.json`

### 6.3 风险：日志和报告口径漂移

控制方式：

- 先冻结合同文档
- 报告顶层回写日志路径
- 新事件新增前先补合同

## 7. 验收标准

满足以下条件即可进入代码验收：

1. 运行中可以直接打开 `benchmark/reports/runs/<run_id>/runtime.log`
2. 出现超时或中断时，日志中能定位到最后一个请求事件
3. `ask_user` 多轮场景里能看到用户模拟 agent 的内部行为
4. 日志中的 `摘要` 不是人工硬编码结果，而是模板自动生成
5. 报告顶层能索引到主日志，并且主日志能继续索引到同目录下的原始证据
