# CRS Agent 需求文档

## 1. 文档目的

本文档基于当前仓库实现，整理 CRS Agent 项目的业务需求、产品范围、角色分工、核心流程、功能清单、非功能要求与阶段边界。

本文档描述的是“当前项目已经实现或明确纳入当前工程边界的能力”，不是未来规划总表。凡是仍处于预留目录、兼容占位或后续演进阶段的内容，会在文中明确标注。

## 2. 项目概述

CRS Agent 是一个面向汽车诊断与维修支持场景的多端系统，目标是为维修技师、服务人员和后台运营人员提供统一的资料检索、故障诊断、维修问答、参数查询、图片证据识别、交互澄清与运行评测能力。

当前项目由以下部分组成：

- 用户前端：`frontend/user`
- 管理后台：`frontend/admin`
- 业务后端：`backend`
- 本地引入的 `pydantic-ai` 源码：`pydantic-ai`
- 评测与数据集资产：`benchmarks`、`docs/benchmark`
- 开发期自动编程引擎：`backend/app/coding_engine`

## 3. 建设目标

### 3.1 业务目标

- 提供统一聊天入口，支持资料检索、故障诊断、维修问答、参数查询等多种业务。
- 支持图片上传，把图片中的故障码、车型、诊断仪文字等信息转成结构化证据。
- 支持在信息不足时通过统一 `ask_user_question` 机制中断并恢复流程。
- 让前端能渲染结构化结果，而不是只能显示纯文本。
- 为后台提供配置、日志、反馈和 Benchmark 管理能力。

### 3.2 工程目标

- 以新的 `FastAPI + Agent Runtime` 结构承载业务，不直接依赖旧项目主链路。
- 对旧项目已有能力采取“复制实现或 adapter 包装”的接入方式。
- 让运行态上下文、消息历史、延迟工具状态、图片证据都能跨轮次复用。
- 保留兼容旧前端协议和旧后端依赖的接入层，降低迁移成本。

## 4. 角色与使用场景

### 4.1 一线用户

主要使用用户端页面，发起资料检索、故障诊断、维修问答、参数查询和图片识别请求。

典型场景：

- 搜索电路图、针脚定义资料、维修手册
- 输入故障码并获取 ECU 候选和诊断结果
- 上传诊断仪、铭牌、ECU、现场图片辅助判断
- 追问维修经验、原理、常见排查步骤
- 查询具体 ECU 针脚、电压、定义值

### 4.2 管理员 / 运营人员

主要使用管理后台进行配置、数据和运行观察。

典型场景：

- 查看服务健康和业务量
- 管理品牌/车系/型号等澄清维度
- 查看任务日志、运行轨迹和工具调用
- 查看用户评分和反馈内容
- 执行资料检索 Benchmark，导出测试结果
- 调整系统配置并刷新缓存

### 4.3 开发人员

主要使用后端与开发期编程引擎能力。

典型场景：

- 本地启动后端与双前端联调
- 调整 Agent Runtime、域服务和适配层
- 使用 LangGraph Studio 驱动 `coding_engine`
- 分析 Benchmark 报表和失败样例

## 6. 核心业务需求

### 6.1 统一聊天入口

系统必须提供统一聊天接口，接收文本消息、上下文、会话标识和补充回答。

需求点：

- 支持同步接口返回结构化响应
- 支持 SSE 流式返回 `start/hint/chunk/done/error`
- 支持 `session_id` 维持会话连续性
- 支持 `ask_user_answer + tool_call_id` 恢复上一轮暂停流程
- 支持客户端生命周期检查，用于显式切换话题

### 6.2 资料搜索能力

系统必须支持面向资料类问题的结构化资料搜索工作流。

需求点：

- 能识别资料检索类请求并进入文档搜索工作流
- 支持基于图片证据增强查询词
- 支持对搜索结果做歧义分析
- 支持品牌、车系、型号等维度澄清
- 支持多轮澄清后重新搜索并收敛
- 支持输出结构化文档结果和无结果提示
- 支持兼容旧 `/search` 接口

### 6.3 参数查询能力

系统必须支持面向 ECU 针脚、引脚定义、电压期望值等问题的结构化参数查询。

需求点：

- 支持从本地结构化缓存中匹配资料源和参数行
- 支持使用 LLM 对自然语言问题做规范化理解
- 支持资料源歧义和参数行歧义的 ask-user 澄清
- 支持返回结构化参数卡片
- 支持后台配置开关控制是否启用
- 支持启动时从外部库同步到本地索引

### 6.4 故障诊断能力

系统必须支持故障码诊断相关的外部能力接入。

需求点：

- 支持根据故障码查询 ECU 候选
- 支持用户选择 ECU 后继续诊断
- 支持故障码图片识别兼容接口
- 支持批量 ECU 查询与批量诊断报告接口
- 当外部诊断服务不可用时，系统不能阻塞整个会话

### 6.5 维修知识问答能力

系统必须支持基于本地 Excel 维修知识库的标题召回、上下文加载和面向用户的维修建议生成。

需求点：

- 从本地 Excel 装载维修知识条目
- 先返回标题候选，再按 entry id 加载正文上下文
- 对维修类问题在回答前先判断信息是否足够
- 在信息不足时生成结构化维修追问表单
- 用户补充后继续同一推理链路
- 对已加载的维修知识来源提供可选展示接口

### 6.6 图片证据识别能力

系统必须支持上传图片并提取结构化证据。

需求点：

- 支持 1 到 3 张图片上传
- 支持校验文件类型与单张大小限制
- 支持识别故障码、诊断仪文字、车型线索、可见文本
- 将识别结果写入聊天上下文与会话上下文
- 支持图片识别可用性检测接口
- 兼容旧故障码图片识别接口

### 6.7 ask-user 统一澄清与恢复

系统必须以统一机制处理中途向用户补充询问的场景。

需求点：

- 所有业务都通过 `ask_user_question` 发起追问
- 追问必须带 `tool_call_id`
- 后端必须保存恢复所需的状态快照
- 前端必须能渲染文本输入、选项选择和结构化表单
- 用户回答后必须能恢复原工作流，而不是从头开始

### 6.8 用户端交互需求

用户端必须支持以下交互：

- 文本输入
- 流式消息展示
- 资料结果卡片与文档查看
- 参数查询卡片
- ask-user v2 表单
- 故障诊断和 ECU 选择卡片
- 图片上传与识别卡片
- 建议问题和反馈卡片
- 阿里云语音能力接入

### 6.9 管理后台需求

管理后台必须支持以下功能：

- 管理员登录与密码修改
- 仪表盘总览
- 维度值新增、修改、删除、缓存刷新
- 配置列表、分类查看、批量更新、刷新缓存
- 日志列表、统计、导出、详情查看
- 用户反馈列表、统计、详情查看
- DocSearch Benchmark 数据集、运行、暂停、继续、报表导出

### 6.10 评测与报表需求

系统必须支持资料搜索评测闭环。

需求点：

- 从数据集启动评测 run
- 支持不同 track：`production_flow`、`raw_retrieval`、`final_list`
- 记录每个 case 的问题、图片、规划查询、候选结果、命中情况
- 输出 `status.json`、`report.json`、`predictions.jsonl`、`events.jsonl`、`failures.csv`
- 支持导出 Excel 报告

## 7. 关键业务流程

### 7.1 聊天主流程

1. 前端发送 `ChatRequest`
2. 后端构建 request 级 runtime deps
3. 解析 token、用户身份、图片证据、会话上下文
4. 做意图路由
5. 优先判断是否进入 doc_search 或 parameter_query 专门工作流
6. 如果未命中专门工作流，则进入主 Agent Loop
7. 若中途需要追问，返回 `ask_user`
8. 若获得最终结果，返回文本、文档、参数卡片或错误

### 7.2 文档搜索流程

1. 识别为资料检索需求
2. 规划查询词，必要时利用图片证据增强
3. 执行搜索
4. 做有效性和歧义分析
5. 若需澄清，生成 `ask_user`
6. 用户选择后按快照恢复并重新搜索
7. 直到收敛为文档结果或无结果消息

### 7.3 参数查询流程

1. 识别为参数查询需求
2. 解析选中资料源、参数行和目标字段
3. 用本地索引匹配资料源与参数行
4. 若存在歧义，返回 `ask_user`
5. 用户补充后继续查询
6. 匹配成功则返回参数卡片

### 7.4 维修问答流程

1. 识别为维修问答类请求
2. 进入 repair gate 判断是否信息充足
3. 使用维修知识标题和正文工具沉淀证据
4. 信息不足则 ask-user
5. 信息足够则规划回答框架并生成最终建议

### 7.5 图片识别流程

1. 前端上传图片
2. 后端校验数量和大小
3. 调用多模态模型提取结构化信息
4. 识别结果进入 `context.image_evidences`
5. 后续意图路由、资料检索、故障诊断和参数查询可复用该证据

## 8. 数据与状态需求

### 8.1 会话级状态

系统必须保存：

- message history
- deferred state
- case context
- request trace

### 8.2 存储策略

默认使用 Redis；Redis 不可用时允许回退到本地 `.data/` 文件存储，以保证开发环境不中断。

### 8.3 后台数据

系统必须在 MySQL 中维护以下业务数据：

- 文档与物理文件索引
- 外部文件 URL 映射
- 维度与维度值
- 用户反馈
- Chat task/run/event 日志
- 参数资料源与参数行
- 管理员账户与系统配置

### 8.4 Benchmark 数据

评测运行必须以文件形式落盘，便于离线分析、导出与回放。

## 9. 非功能需求

### 9.1 可恢复性

- ask-user 恢复必须依赖 `session_id + tool_call_id`
- 文档搜索和参数查询必须能从上轮状态恢复
- 流式中断时支持保存部分内容

### 9.2 可观测性

- 每次请求必须记录 trace 事件
- 后台必须能查看 task、run、event 级日志
- 响应中需要带 `request_id` 用于问题追踪

### 9.3 可配置性

- 模型、外部地址、超时、开关、阈值、缓存策略需可配置
- 敏感配置在后台展示时需支持跳过原值回填

### 9.4 兼容性

- 保留旧 `/search`、`/legacy/*`、图片诊断兼容接口
- 用户端继续使用 `/chat/api` 作为统一代理前缀

### 9.5 性能与体验

- 支持 SSE 流式输出，降低长请求等待感
- 资料搜索与工具链路支持更长超时配置
- 图片上传需限制大小和数量

### 9.6 安全

- 用户端请求通过 `x-app-token` 传递业务 token
- 管理后台使用 JWT
- 管理端部分操作要求管理员权限

## 10. 当前实现边界与约束

### 10.1 明确已落地

- 新后端已不是骨架，已承载真实运行时
- 用户端与管理端均可本地启动并连接后端
- 文档搜索、参数查询、维修问答、图片证据和反馈闭环已接入
- Benchmark 与 coding engine 已具备基础可用实现

### 10.2 仍存在的现实约束

- 资料搜索和部分业务能力仍依赖旧系统适配层
- Mem0 默认关闭
- 小程序前端未迁移
- 前端部分 README 仍保留早期说明，与当前实现并不完全一致
- 多能力统一协同回答仍处于演进态，当前更多是“专门工作流 + 主 Agent”的组合架构

## 11. 验收视角

若以当前项目为准，阶段性验收应至少覆盖：

- 用户端能发起文本、图片、流式聊天
- 文档搜索可多轮澄清并返回结果
- 参数查询能返回结构化参数卡片
- 维修问答能在必要时 ask-user 并恢复
- 图片识别结果可注入聊天上下文
- 管理后台可登录、查看日志、反馈、配置和 Benchmark
- Benchmark 可启动、暂停、继续并导出报表
- coding engine 可在 LangGraph Studio 中运行并落盘
