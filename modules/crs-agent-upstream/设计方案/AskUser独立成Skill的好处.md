# AskUserQuestion 独立成 Skill 的好处

> 编写日期：2026-03-19
> 状态：方案讨论阶段

## 一、背景

在 Agent Loop 架构中，AskUserQuestion（向用户主动提问）有两种定位方式：

- **方案 A**：作为系统能力，内建在框架中，任何 tool 可在执行过程中触发
- **方案 B**：作为独立 skill，和 doc_search、dtc_diagnosis 等并列，由 LLM 自主决定何时调用

**结论：方案 B（独立 skill）更优。**

## 二、核心论点：tool 不应该负责问用户

以一个实际场景说明两种方案的差异。

**场景：用户输入"发动机抖动"，doc_search 搜出多个品牌的结果，需要用户澄清。**

### 系统能力方案的流程

```
LLM 调用 doc_search("发动机抖动")
→ doc_search 执行中发现结果涉及多个品牌
→ doc_search 内部触发框架的 NeedUserInput 机制
→ 框架中断 doc_search 的执行，将问题推给前端
→ 用户回答后，框架恢复 doc_search 从中断点继续执行
```

**问题：**

1. **tool 函数被挂起再恢复**，工程上极其复杂，需要协程级别的挂起恢复机制
2. **doc_search 的职责被污染**——一个搜索工具里混入了用户交互逻辑
3. **每个可能需要澄清的 tool 都要实现类似的中断逻辑**，代码重复且耦合框架

### 独立 skill 方案的流程

```
LLM 调用 doc_search("发动机抖动")
→ doc_search 正常执行，正常返回："找到多个匹配——东风天锦(3篇)、解放J6(2篇)"
→ LLM 看到结果涉及多个品牌，判断需要澄清
→ LLM 调用 ask_user_question("您要查哪个品牌？", options=["东风天锦", "解放J6"])
→ Loop 中断，问题推给前端
→ 用户回答 "东风天锦"
→ Loop 恢复，LLM 调用 doc_search("发动机抖动", brand="东风天锦")
→ doc_search 返回精确结果
→ LLM 综合生成最终回复
```

**优势：doc_search 只管搜索并返回结果，"够不够、要不要问用户"由 LLM 判断。**

## 三、独立 skill 的五个好处

### 1. Tool 保持纯函数

每个 skill 只做一件事：doc_search 搜文档，dtc_diagnosis 查故障码，ask_user_question 问用户。没有任何 tool 需要了解用户交互机制，输入进去输出出来，职责清晰。

### 2. 符合 Agent Loop 的设计哲学

Agent Loop 的核心思想是 **LLM 是编排者**。"什么时候该问用户"是一个编排决策，应该由 LLM 做，而不是由某个 tool 在执行过程中自行判断。

这和人类团队协作的逻辑一样——你让一个技术员去查资料，他查完告诉你结果。至于要不要找客户确认，是你（负责人）来决定的，不是技术员自己去找客户。

### 3. 实现简单，没有挂起恢复的复杂度

系统能力方案需要处理"tool 执行到一半被中断，用户回答后从断点恢复"，这涉及函数级协程挂起，状态序列化极其困难。

独立 skill 方案中，每次 tool 调用都是完整执行、完整返回。中断只发生在 tool 之间（loop 的轮次间隔），不发生在 tool 内部。Pydantic AI 的 DeferredTools 机制天然支持这种模式。

### 4. 用户交互样式集中管理

所有面向用户的交互（选项卡、多选、数值输入、纯文本）只在 ask_user_question 一个 skill 里定义。前端也只需对接这一个交互协议。

如果做成系统能力散落在框架各处，各个 tool 各自定义自己的问题格式，前端需要处理多种来源的交互请求，维护成本高。

### 5. 易于扩展交互形式

未来如果需要新的交互类型（比如图片上传、语音输入、确认操作），只需修改 ask_user_question 这一个 skill 的定义和前端对应的渲染逻辑，其他 skill 完全不受影响。

## 四、一个完整的多步诊断示例

**用户输入："发动机抖动，帮我查一下什么问题"**

```
Loop 第1轮：
  LLM 推理 → 信息不足，不知道工况
  LLM 调用 ask_user_question(
    question="发动机在什么状态下抖动？",
    options=["冷启动", "怠速", "加速中", "匀速行驶"],
    allow_free_input=true
  )
  → Loop 中断，等待用户

用户回答："冷启动"
  → Loop 恢复

Loop 第2轮：
  LLM 推理 → 冷启动抖动，可能和喷油器或预热塞有关，先搜文档
  LLM 调用 doc_search("冷启动抖动 喷油器 预热塞")
  → 返回："找到多个匹配——东风天锦(3篇)、解放J6(2篇)"

Loop 第3轮：
  LLM 推理 → 结果涉及多品牌，需要确认
  LLM 调用 ask_user_question(
    question="您的车型是？",
    options=["东风天锦", "解放J6"]
  )
  → Loop 中断，等待用户

用户回答："东风天锦"
  → Loop 恢复

Loop 第4轮：
  LLM 调用 doc_search("冷启动抖动 喷油器 预热塞", brand="东风天锦")
  → 返回具体维修资料

Loop 第5轮：
  LLM 调用 dtc_diagnosis("冷启动抖动")
  → 返回可能相关的故障码列表

Loop 第6轮：
  LLM 综合文档搜索结果 + 故障码信息，生成完整诊断建议
  → 输出最终回复，Loop 收束
```

全程 doc_search 和 dtc_diagnosis 不知道用户交互的存在，它们只负责搜索和诊断。所有的"问用户"决策都由 LLM 发起，通过 ask_user_question 统一执行。
