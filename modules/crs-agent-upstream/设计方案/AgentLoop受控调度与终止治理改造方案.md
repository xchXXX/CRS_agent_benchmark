# AgentLoop 受控调度与终止治理改造方案

> 编写日期：2026-03-27
> 状态：方案设计

## 1. 文档目标

这份文档回答一个非常具体的问题：

当前项目已经把多类工具接入到了 AgentLoop 中，模型可以在一次请求内部自主选择工具、读取结果并继续推理。但在工具变多之后，系统开始出现一个新的工程问题：

- 模型会把“继续调用工具”当成默认安全动作
- 即使已经拿到了足够的信息，也可能继续外部交互
- 即使程序已经尽量要求它“一次性列清需要的信息”，仍然可能进入重复检索、重复追问、重复外部调用

这类问题不是单纯靠 prompt 能彻底解决的，而是典型的 loop harness 设计问题。

本文的目标不是直接给出代码，而是形成一份清晰的改造方案，明确：

1. 当前实现已经做到哪里
2. 当前失控问题的根因是什么
3. 应该如何把自由 loop 改造成受控 loop
4. 需要新增哪些状态、规则和决策层
5. 应该按什么顺序分阶段落地

## 2. 当前实现现状

从最新代码看，当前 AgentLoop 并不是完全无约束的自由循环，已经有几层基础能力，但这些能力还不足以彻底解决“反复调用工具”的问题。

### 2.1 已经具备的基础

当前代码已经有以下正向基础。

#### 2.1.1 请求前置分流

系统已经有显式的意图分流器，会先把请求分到：

- `DOC_SEARCH`
- `PARAM_QUERY`
- `FAULT_DIAGNOSIS`
- `GENERAL_CHAT`

这意味着系统已经不是“所有请求直接进一个完全自由的大 Agent”，而是具备了第一层业务分流。

#### 2.1.2 文档搜索已经是确定性 workflow

`doc_search` 已经不是由主模型自由发挥，而是走外部工作流：

- 恢复澄清状态
- 执行搜索
- 分析歧义
- 必要时返回 ask-user
- 否则直接返回文档结果

这条链路已经具备了“业务闭环 + 显式终止”的特征，是当前项目中最接近受控状态机的实现。

#### 2.1.3 维修知识已经开始做两段式 gate

维修知识链路已经出现了一个很重要的演进方向：

- `repair_gate_agent` 负责判断“信息是否足够回答”
- `repair_renderer_agent` 负责在确认可答后输出最终答复

这已经不是单纯让一个模型既决定是否继续搜、又决定是否继续问、还负责最终渲染，而是开始把“判断”与“输出”拆开。

#### 2.1.4 已有基础 loop guard

当前系统已经有每请求级别的 `LoopGuard`，并在工具调用前执行硬拦截。现有 guard 可以控制：

- 最大工具调用总次数
- 同一工具最大重复次数
- 同参调用最大重复次数

这说明当前架构已经接受了一个关键事实：

“不能完全依赖模型自觉停止，必须由外部运行时做确定性约束。”

#### 2.1.5 已有共享 case context 和上下文压缩

当前系统已经不是把全部原始 tool output 无限制地回灌给模型，而是开始维护共享 `case_context`，包括：

- 已确认槽位
- 最近证据摘要
- pending action
- 受控长度的上下文摘要

这为后续演进到真正的 working state 奠定了基础。

### 2.2 当前仍然存在的关键缺口

虽然已有基础，但主问题依然存在，原因在于现在的控制仍然偏轻，且主要发生在“工具调用前的硬拦截”这一个点上。

当前仍然存在以下缺口。

#### 2.2.1 主 agent 仍然是开放式运行

当前通用主流程仍然是把请求直接交给主 agent 运行。也就是说，在 `doc_search` 和部分 `repair_knowledge` 之外，主 agent 仍然可以：

- 决定调什么工具
- 决定调几次
- 决定什么时候停止

虽然有 guard，但 guard 更像“撞墙保护”，还不是“路线规划器”。

#### 2.2.2 现有 guard 只有“次数”，没有“价值”

当前系统能判断：

- 调了几次
- 是否同参重复

但还不能判断：

- 这次调用是否带来了新信息
- 当前是否已足够回答
- 当前是否缺少关键槽位
- 是否存在更便宜的替代动作
- 继续外部调用是否还有边际收益

也就是说，当前系统能约束“量”，还不能约束“是否值得继续”。

#### 2.2.3 guard 超限后的策略过于生硬

当前 guard 命中后，主要策略还是直接返回错误。

这能阻止失控，但体验并不理想，因为更合理的行为通常应该是以下三种之一：

- 输出当前最佳答案
- 改为 ask-user 收集关键缺失信息
- 明确告知用户信息不足并结束本轮

直接报错更像“防爆丝”，不是“收敛机制”。

#### 2.2.4 当前 case context 还不是完整 working state

现在的 `case_context` 已经能存储证据和槽位，但还缺少真正驱动 loop 决策的状态字段，例如：

- 当前缺失的关键槽位
- 已尝试动作列表
- 每次动作的信息增量结论
- 连续无增量次数
- 剩余预算
- 当前候选结论
- 是否已达到回答阈值

没有这些状态，调度层就很难做真正的“停止审查”。

#### 2.2.5 工具在主 agent 视角下仍然过于平权

当前多个工具基本都直接暴露给主 agent：

- 参数查询
- 维修知识
- 故障诊断
- ask-user

这会让模型自然倾向于把工具当作低成本试探手段，而不是把它们视为受控资源。

## 3. 当前问题的本质

这个问题的本质不是“模型笨”，而是 loop 设计过于开放。

对于模型来说，在不确定时，“继续查一下”“再调用一次”“再多拿一点信息”通常是一个非常安全的默认动作。因为：

1. 继续调用工具，通常不会立刻被判错
2. 停止并直接回答，反而承担更高的答错风险
3. 如果没有强终止条件，模型自然会偏向继续探索

这意味着：

- prompt 可以降低问题概率
- 但 prompt 不能从根上解决问题
- 根治必须依赖外部、确定性的 harness 控制

一句话概括：

当前系统的问题不是工具太多，而是“是否继续调用工具”仍主要由模型决定。

## 4. 改造原则

这次改造应该遵循以下原则。

### 4.1 不追求“模型自己学会停止”

不要把“会不会停”当成模型能力问题。

应明确拆成两层：

- 模型层：提出下一步候选动作
- 调度层：决定动作是否允许执行、是否需要降级、是否必须结束

### 4.2 不推翻现有 AgentLoop 底座

当前已有的这些能力需要保留：

- 会话连续性
- ask-user 暂停与恢复
- 消息历史持久化
- 流式输出
- 现有业务适配器

本次改造不是重写整个系统，而是在现有运行时外层加一层受控调度能力，并把部分高风险业务迁移为显式 workflow。

### 4.3 优先增强“受控性”，而不是优先增强“智能性”

在现阶段，最有价值的不是让模型更会思考，而是让系统：

- 更稳定地停下来
- 更少地重复查
- 更早地 ask-user
- 更明确地在预算内收敛

### 4.4 能规则化的地方优先规则化

尤其在当前领域中，很多判断其实是可以外部规则化的：

- 是否缺关键槽位
- 是否属于高成本外部调用
- 是否已经重复同参查询
- 是否连续无信息增量
- 是否已具备最小回答条件

这些不应该优先交给主模型自由判断。

## 5. 目标架构

建议把当前“开放式 AgentLoop”演进为“有预算的受控状态机”。

目标流程如下：

用户请求
→ 意图分流
→ 领域工作流/主代理产生候选动作
→ 调度层审查动作
→ 执行动作
→ 抽取信息增量
→ 更新 working state
→ 判断是否继续 / ask-user / 最终回答 / 收敛失败

其中最核心的变化是：

### 5.1 主模型不再直接拥有“继续循环”的最终决定权

模型只负责输出候选动作，例如：

- 调工具
- ask-user
- 直接回答
- 结束并说明不足

但是否真的执行，由外部调度层决定。

### 5.2 所有循环必须有显式终止条件

系统应保证每条可循环链路都具备明确 stop condition，而不是依赖模型“看起来差不多了”。

### 5.3 不再把原始工具结果无限堆进上下文

模型看到的应主要是整理后的工作状态，而不是越来越长的原始消息链。

## 6. 建议的核心设计

### 6.1 把 loop 改为“Decision -> Guard -> Execute -> Update”

每轮运行固定分成四个阶段。

#### 阶段 A：Decision

模型只输出候选动作，不直接执行。候选动作至少包括：

- 动作类型
- 动作原因
- 预期补充的槽位或证据
- 失败后是否应停止

例如：

```json
{
  "action": "ask_user",
  "reason": "缺少车型和 ECU 型号，继续查询命中率低",
  "expected_gain": "high",
  "fills_slots": ["model", "ecu_model"],
  "stop_if_fail": true
}
```

#### 阶段 B：Guard Review

调度层对候选动作做确定性审查，审查内容包括：

- 是否超预算
- 是否与上一步重复
- 是否缺少执行前提
- 是否存在更便宜替代动作
- 是否已经达到回答阈值
- 是否应改为 ask-user

只有通过审查的动作才允许执行。

#### 阶段 C：Execute

执行被批准的动作。

#### 阶段 D：Update

不回灌原始全文，而是把这次执行结果整理成结构化增量，更新 working state。

### 6.2 引入全局预算体系

每个请求都应有明确预算。

建议最小预算集合如下：

- 最大 loop 步数
- 最大工具调用总数
- 最大外部工具调用数
- 同类工具最大重复次数
- 同参最大重复次数
- 最大 ask-user 次数
- 最大连续无增量次数
- 最大耗时预算
- 最大 token 预算

超预算后不再继续让模型选择动作，而应直接进入收敛分支。

### 6.3 引入动作分级

不要让主 agent 把所有工具视为同成本动作，建议把动作分成三层：

#### L1：低成本、本地、安全动作

- case context 读取
- 槽位归一化
- 本地索引检索
- 本地参数查询

#### L2：中成本业务动作

- 维修知识标题检索
- 本地维修知识正文加载
- 本地诊断信息整合

#### L3：高成本或外部交互动作

- 外部文档搜索
- 第三方 API
- 写操作
- 需要明确人机切换的交互动作

调度层应优先尝试低成本动作，只有低成本动作不足时才升级。

### 6.4 引入“信息增量”判定

每次动作执行后，系统要明确判断这次动作有没有带来新增信息。

可采用的判定方式包括：

- 是否新增关键槽位
- 是否新增可用证据
- 是否显著缩小候选范围
- 是否改变当前候选答案排序
- 是否完成了某个 pending action

连续两次无信息增量时，应强制收敛，而不是继续调用工具。

### 6.5 引入关键槽位 gating

当缺失的是关键槽位时，系统应禁止继续盲目检索，直接切换到 ask-user。

示例：

- 资料搜索缺少品牌、车系、资料类型
- 故障诊断缺少故障码或 ECU
- 参数查询缺少明确 pin、source 或 ECU
- 维修知识问答缺少关键现象、工况、报码情况

这类问题继续调用工具的收益很低，应该优先 ask-user。

### 6.6 引入回答阈值 `answer_ready`

系统需要一个外部的、明确的“可回答”判定，而不是完全依赖主模型说“我觉得差不多了”。

建议 `answer_ready` 至少考虑：

- 关键槽位是否齐全
- 是否已有至少一条高置信证据
- 是否仍存在未解决冲突
- 用户问题是否已被覆盖
- 再继续调用工具的边际收益是否低

满足阈值后，应禁止继续搜索，直接进入最终回答。

## 7. 建议的 working state

当前 `case_context` 应逐步演进为真正的 working state。

建议目标结构至少包含以下内容：

```json
{
  "session_id": "xxx",
  "task_type": "repair_knowledge",
  "confirmed_slots": {
    "brand": "东风",
    "series": "天锦",
    "fault_code": "P20EE"
  },
  "missing_slots": ["ecu_model", "working_condition"],
  "attempted_actions": [
    {
      "action": "lookup_repair_knowledge_titles",
      "args_signature": "abc123",
      "result_summary": "命中 3 条候选标题，已加载 1 条正文",
      "info_gain": "medium"
    }
  ],
  "candidate_answer": null,
  "pending_action": {
    "type": "ask_user",
    "reason": "缺少工况"
  },
  "budgets": {
    "loops_left": 3,
    "tool_calls_left": 2,
    "external_calls_left": 1,
    "ask_user_left": 1
  },
  "no_gain_streak": 1,
  "answer_ready": false
}
```

这个 working state 的意义不是“把历史搬个家”，而是让调度层能依据状态做确定性决策。

## 8. 结合当前业务线的改造建议

### 8.1 文档搜索

文档搜索目前已经是最接近目标形态的能力，应继续保持“外部闭环 workflow”设计，不要回退到自由 agent。

后续建议补充：

- clarify 轮次预算
- 连续无增量终止
- 明确的“无有效结果”收敛策略

### 8.2 参数查询

参数查询天然适合做“强约束技能”而不是自由探索链路。

建议：

- 对参数问题优先走显式 skill
- 若缺 ECU/source/pin 等关键槽位，优先 ask-user
- 若已命中结构化结果，则直接进入 answer-ready
- 禁止参数查询在同轮内无限重试

### 8.3 故障诊断

故障诊断应继续强化为明确的业务状态机。

建议按以下阶段拆分：

- 故障码识别
- ECU 判定
- 诊断执行
- 结果整合

当 ECU 不明确时，只允许 ask-user，不允许模型继续尝试其它高成本动作。

### 8.4 维修知识

维修知识目前已经有 `gate + renderer` 雏形，是最适合继续深化的链路。

下一步建议：

- 把 gate 从 prompt 约束升级为真正的 guard review
- 显式维护缺失字段清单
- 对“继续查标题/继续读正文”引入 no-gain 审查
- 若缺关键信息，则直接 ask-user
- 若已达到回答阈值，则禁止再调用工具

## 9. 最小可落地改造方案

为了控制改造风险，建议先做一版最小闭环，不追求一步到位。

### 9.1 Phase 1：补足当前 guard 能力

目标：先把“明显失控”控制住。

建议新增：

- 最大 ask-user 次数
- 最大外部调用次数
- 连续无增量次数限制
- 更强的同参标准化去重

本阶段暂不引入复杂状态机，只增强 guard 能力。

### 9.2 Phase 2：把错误型 guard 改成收敛型 guard

目标：guard 命中后不再优先报错，而是优先收敛。

建议改成三类收敛结果：

- `best_effort_answer`
- `ask_user_required`
- `insufficient_information`

这一步会显著改善用户体验。

### 9.3 Phase 3：扩展 case context 为 working state

目标：让调度层拥有真正能驱动停止决策的状态。

建议新增：

- missing slots
- attempted actions
- no-gain streak
- remaining budgets
- candidate answer
- answer-ready flag

### 9.4 Phase 4：为高风险业务接入显式 guard review

优先顺序建议如下：

1. `repair_knowledge`
2. `fault_diagnosis`
3. `param_query`
4. 通用主 agent

原因是前两者最容易出现“还能继续搜”的行为，而且业务收益也最高。

### 9.5 Phase 5：逐步收缩自由主 agent 的工具面

最终目标不是让一个主 agent 直接看到所有底层工具，而是：

- 主 agent 看到的是高层 skill
- skill 内部再决定具体调用哪些底层工具
- skill 运行仍由外部 harness 监管

这样可以从根上降低无意义工具试探。

## 10. 目标后的运行形态

改造完成后，系统希望达到的不是“永不调用多个工具”，而是“只有在值得时才继续调用”。

理想状态下，一轮请求的推进逻辑应是：

1. 先判断属于哪类任务
2. 再判断是否缺关键槽位
3. 缺关键槽位则 ask-user
4. 不缺则优先执行低成本动作
5. 每次执行后判断是否有增量
6. 达到回答阈值则立即结束
7. 若无增量连续发生则停止
8. 若预算耗尽则收敛输出当前最优结果

这时模型仍然重要，但它负责的是“提出下一步候选动作”，不再负责“决定系统能否继续消耗资源”。

## 11. 预期收益

完成这轮改造后，预期会得到以下收益。

### 11.1 工具滥调用显著下降

尤其是：

- 同参重复调用
- 明知缺槽位仍继续搜
- 多轮无增量检索
- 明明已够回答却继续查

### 11.2 用户体验更稳定

系统会更快进入以下几类清晰结果：

- 直接回答
- 一次性 ask-user
- 结构化说明信息不足

而不是在请求内“磨很多步后突然报错”。

### 11.3 成本和时延更可控

预算、无增量终止和高成本动作 gating 会直接减少：

- 外部 API 次数
- 不必要模型 token 消耗
- 流式等待时长

### 11.4 架构更适合后续多能力协同

受控调度层建立之后，未来要做多能力联合编排时，就不会演变成“工具更多，失控更严重”，而会建立在统一预算和统一工作状态之上。

## 12. 风险与注意事项

### 12.1 不要一次性把所有链路都改成状态机

更合理的方式是先在最容易失控、收益最大的链路上试点，再逐步推广。

### 12.2 不要只加强 prompt 而不改 runtime

如果只写更多规则到 system prompt，系统短期可能看起来好一点，但问题不会从根上消失。

### 12.3 不要把 guard 做成纯阻断器

guard 如果只有“禁止”，没有“阻断后怎么办”，用户体验会很差。

必须同时设计：

- 被拦截后的收敛策略
- 该何时回答
- 该何时 ask-user

### 12.4 不要让 working state 重新膨胀成另一份原始历史

working state 的目标是“决策摘要”，不是把原始消息链换个结构名后继续无限累计。

## 13. 明确结论

对当前项目而言，最稳的方向不是追求一个“会自己想明白何时停止”的全能 agent，而是构建一个：

- 有预算
- 有状态
- 有动作审查
- 有信息增量判定
- 有回答阈值
- 有业务技能边界

的受控 AgentLoop 系统。

一句话概括这次改造的核心方向：

让 LLM 负责提出动作，让程序负责决定是否允许动作。

这不是对现有 AgentLoop 的否定，而是当前项目从“能跑起来”走向“可控、可收敛、可扩展”的必经阶段。
