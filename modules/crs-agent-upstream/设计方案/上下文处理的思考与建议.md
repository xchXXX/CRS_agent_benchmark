# 上下文处理的思考与建议

> 编写日期：2026-03-19
> 状态：方案讨论阶段

## 一、我们面对的上下文问题

在 Agent Loop 架构下，上下文管理比单跳架构复杂得多。单跳架构一次请求一次回复，上下文就是一轮对话。Agent Loop 中，一次用户请求可能触发 3-5 轮 tool call，每轮都会往消息链里追加 tool call + tool return，上下文膨胀速度远超单跳模式。

但基于当前判断——用户 session 不会太长，暂不做会话历史功能——上下文问题的优先级和范围可以收窄。

## 二、需要处理的三个层次

| 层次 | 问题 | 优先级 | 当前判断 |
|------|------|--------|---------|
| 单轮 loop 内 | 一次用户请求触发多次 tool call，消息链膨胀 | 高 | 必须处理 |
| 单 session 内多轮对话 | 用户连续问多个问题，历史累积 | 中 | session 不长，简单策略即可 |
| 跨 session 长期记忆 | 记住用户车型、历史故障等 | 低 | 暂不做 |

## 三、单轮 loop 内的上下文控制

这是当前最需要关注的。一次用户请求进入 Agent Loop 后，LLM 可能连续调用多个 tool，每个 tool 的返回值都会写入消息链。

### 问题场景

用户问"东风天锦 P0087 帮我查一下"，loop 内部可能：

```
tool call: dtc_diagnosis("P0087") → 返回 2000 tokens 的诊断报告
tool call: doc_search("P0087 东风天锦") → 返回 3000 tokens 的文档片段
tool call: ask_user_question("需要查电路图吗？") → 中断等待
```

仅一轮 loop 就可能产生 5000+ tokens 的 tool 返回值。如果 LLM 继续推理又调了几个 tool，很快逼近上下文窗口。

### 应对策略

**1. 控制 tool 返回值的长度**

这是最直接有效的手段。每个 tool 返回时做截断和摘要，而不是把原始数据全部灌进上下文。

- doc_search：返回摘要 + 文档链接，不返回全文
- dtc_diagnosis：返回关键诊断结论，不返回完整报告原文
- 设定每个 tool 返回值的 token 上限（建议 1000 tokens）

**2. UsageLimits 硬限制**

通过 Pydantic AI 的 UsageLimits 设定 loop 的最大轮次和 token 预算，防止失控。

**3. System prompt 引导收束**

在 system prompt 中明确告诉 LLM：优先用最少的 tool 调用解决问题，信息足够时立即生成回复，不要过度探索。

## 四、单 session 内多轮对话的处理

用户在一个 session 里可能连续问 5-10 个问题。每个问题的 loop 结束后，消息链作为历史传入下一轮。

### 当前判断

session 不会太长，不需要复杂的压缩机制。但仍需要基本的控制，防止极端情况。

### 建议策略：滑动窗口

保留最近 N 轮完整对话（建议 N=5），更早的对话直接丢弃。

理由：
- 维修诊断场景下，用户的问题通常相对独立（"查个故障码"、"找个电路图"），跨轮依赖不强
- 即使有依赖，最近 5 轮通常足够覆盖上下文
- 实现极其简单，不需要调模型做摘要，零额外成本

### 关键约束

截断时必须保证 tool call 和 tool return 成对出现。不能只截掉一半，否则 LLM 会报错。截断逻辑应以"完整的一轮对话"为最小单位，而不是按消息条数截断。

## 五、跨 session 长期记忆（暂不实施）

记录备忘，未来如果需要可以考虑的方向：

| 方案 | 定位 | 适合场景 |
|------|------|---------|
| Mem0 | 知识点提取引擎 | 从对话中提取用户事实（车型、偏好），跨 session 积累用户画像 |
| OpenViking | 上下文数据库 | L0/L1/L2 分层加载，统一管理记忆/文档/技能，渐进式压缩 |

两者解决不同问题：Mem0 偏"记住用户是谁"，OpenViking 偏"管理所有上下文的加载策略"。当前阶段都不需要引入。

## 六、总结

当前阶段的上下文策略应该是**轻量、务实、够用**：

1. **控制源头**：每个 tool 返回值做截断，不往上下文里灌大量原始数据
2. **限制 loop**：UsageLimits 设定最大轮次和 token 预算
3. **滑动窗口**：session 内保留最近 5 轮完整对话，更早的丢弃
4. **配对检查**：截断时保证 tool call / tool return 成对完整

不需要摘要模型、不需要向量召回、不需要外部记忆库。等业务跑起来后，根据实际的 session 长度和用户行为数据再决定是否引入更复杂的方案。
