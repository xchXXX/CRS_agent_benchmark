# AskUserQuestion 设计方案

> 编写日期：2026-03-19
> 状态：方案讨论阶段

## 一、定位

AskUserQuestion 是 Agent Loop 中的一个特殊 skill，用于在诊断推理过程中向用户主动提问以获取缺失信息。它是系统从"单轮应答"升级为"多轮协作诊断"的关键能力。

## 二、两种提问模式

| 模式 | 触发方式 | 适用场景 |
|------|---------|---------|
| 开放式追问 | LLM 直接在回复文本中提问 | "请详细描述故障现象" 等模糊问题 |
| 结构化提问 | LLM 调用 ask_user_question tool | 需要前端渲染选项卡/按钮的场景 |

开放式追问不需要特殊机制——LLM 输出问题文本，用户下一条消息自然带着回答进入新一轮对话。

**本方案重点设计结构化提问**，因为它涉及 loop 中断恢复和前端 UI 渲染。

## 三、支持的输入类型

结合维修诊断场景，需要支持以下输入类型：

| 输入类型 | 场景举例 | 前端呈现 |
|---------|---------|---------|
| 单选 + 可输入 | "发动机在什么状态下抖动？" → [冷启动, 怠速, 加速中, 其他] | 选项卡 + 底部自由输入框 |
| 多选 + 可输入 | "目前出现了以下哪些症状？" → [抖动, 异响, 冒黑烟, 动力不足] | 可多选的选项卡 + 底部补充输入框 |
| 数值输入 | "请测量针脚3的电压" → 参考范围 4.5-5.5V | 数值输入框 + 单位 + 参考范围提示 |
| 纯文本输入 | "请详细描述故障发生的过程" | 文本输入区域 + 占位提示 |

每个选项包含 `label`（显示文字）和 `description`（补充说明），降低维修技师的理解成本。

## 四、核心交互流程

### 三段式流程：中断 → 持久化 → 恢复

```
用户: "发动机抖动"
    ↓
Agent Loop 启动，LLM 推理
    ↓
LLM 判断信息不足 → 调用 ask_user_question
    ↓
【中断】Loop 终止，返回问题数据给前端
    ↓
【持久化】完整消息链 + 会话状态存入 Redis
    ↓
前端渲染问题卡片（选项 + 自由输入），等待用户操作
    ↓
用户选择/输入后提交
    ↓
【恢复】从 Redis 恢复消息链，用户回答作为 tool 返回值注入上下文
    ↓
Agent Loop 继续推理（LLM 看到完整的问答上下文）
```

### 关键设计点

**用户回答的本质是 tool 返回值。** ask_user_question 的返回和 doc_search、dtc_diagnosis 的返回在消息链中格式完全一致，都是 ToolReturnPart。这意味着上下文管理、历史压缩的逻辑对所有 tool 统一，不需要为 AskUserQuestion 单独设计上下文策略。

唯一区别是**时间维度**——普通 tool 毫秒级返回，AskUserQuestion 可能几分钟甚至不返回，因此需要 Redis 持久化中间状态。

## 五、前端交互设计

### SSE 事件扩展

在现有 SSE 事件流中新增 `ask_user` 事件类型：

```
data: {"type": "ask_user", "question": "...", "input_type": "select",
       "options": [...], "multi_select": false, "allow_free_input": true,
       "input_hint": "...", "tool_call_id": "call_xxx"}
```

前端根据 `input_type` 渲染不同 UI 组件。

### 用户操作路径

1. **选择选项** → 提交选中项
2. **选择选项 + 补充输入** → 提交选中项 + 补充文字
3. **忽略选项直接输入** → 提交自由文字
4. **跳过问题** → 前端统一提供"跳过"按钮，注入"用户跳过此问题"到上下文，LLM 自动调整诊断策略
5. **忽略问题直接问新问题** → 走普通消息流程（见边界情况处理）

### 用户回答格式化

用户回答注入上下文前需格式化为 LLM 可理解的字符串：

- 单选/多选：`"用户选择: 冷启动, 冒黑烟\n用户补充: 偶尔还会熄火"`
- 数值输入：`"用户测量值: 3.2V"`
- 纯文本：`"用户回答: 昨天开始的，每次冷启动都抖"`
- 跳过：`"用户跳过: 暂时无法测量"`

## 六、边界情况处理

### 1. 连续反问

LLM 恢复后可能再次调用 ask_user_question（问了工况又想问车型），流程重新走"中断 → 持久化 → 恢复"。

**限制：** system prompt 中约束最多连续提问 3 次，代码层面兜底。超限后 LLM 基于已有信息给出最佳建议并说明哪些信息缺失。

### 2. 用户不回答反问，直接问新问题

```
LLM: "请问是冷启动还是热车时抖动？"
用户: "P0087 故障码什么意思"  ← 完全换了话题
```

判断逻辑：如果请求类型不是 `deferred_answer`，即使 session 处于"等待回答"状态，也当作新的对话轮次处理。清除 pending 状态，之前的消息链作为普通历史传入。

### 3. 超时清理

用户可能关掉页面不再回来。pending_messages 设 TTL（建议 1 小时），过期自动清理。

## 七、与上下文管理的关系

AskUserQuestion 和上下文管理是同一个问题的两面：

- **上下文管理**解决的是消息链的存储、恢复、压缩
- **AskUserQuestion**只是增加了一种"消息链中间暂停"的场景

设计上下文压缩策略时，需要保证 tool call 和 tool return 成对出现——如果截掉了一个 ask_user_question 的调用但保留了用户回答的返回，LLM 会无法理解上下文。
