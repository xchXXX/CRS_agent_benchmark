# Agent Loop 运行机制与诊断思维框架

> 编写日期：2026-03-20
> 状态：方案讨论阶段

## 一、核心决策：全部走 Agent Loop，放弃快速路径

### 为什么放弃正则/规则引擎的快速路径

现有架构的问题不是慢，而是**浅**。以"EDC17CV44 报了 P01F5"为例：

- 现有架构：正则命中故障码 → 调诊断接口 → 返回报告，结束。用户拿到一份报告，但不知道下一步该测什么、该看哪张电路图、类似案例别人怎么处理的。
- Agent Loop：LLM 拿到故障码后，自主调诊断接口、搜电路图、搜维修案例，综合分析后不仅给报告，还引导用户检测具体针脚、给出正常参考值、附上电路图、追问测量结果以便进一步缩小故障范围。

快速路径的本质是"找到就结束"，但系统的核心价值在于**深度诊断式的交互过程**，不在于省那几秒。全部走 Agent Loop 才能释放这个价值。

## 二、System Prompt 是核心中的核心

没有快速路径后，所有请求都进 loop，LLM 的行为完全由 System Prompt 决定。

### 设计原则

System Prompt 不是 SOP 死流程（"第一步做A，第二步做B"），而是**诊断思维框架**——像老师傅带学徒，教的是思路不是步骤。写成死流程就退化回了现有架构，丧失了 Agent Loop 的灵活性。

### 诊断思维框架（三层结构）

**第一层：信息收集 —— 够不够做判断？**

拿到用户输入后，先评估已有信息是否足够开始诊断：
- 有故障码但不知道车型/ECU → 问用户
- 有故障现象但描述模糊 → 问用户具体工况（冷启动？热车？加速时？）
- 车型和故障都清楚 → 直接进入诊断

这一层对应 ask_user_question skill。关键是不要过度追问——能从已知信息推断的就不要问。

**第二层：多维度调查 —— 不要只查一个方向就停**

围绕问题做全面调查，而不是只返回一个诊断报告：
- 故障码解读 → dtc_diagnosis
- 相关电路图 → search_circuit_diagram
- 维修经验/类似案例 → search_repair_knowledge
- 相关参数正常值 → 参数查询（未来能力）

LLM 应自主判断需要调查哪些维度。不是每次都要调所有 tool，而是根据具体问题决定。

**第三层：综合研判 + 引导检测**

把所有调查结果综合起来，输出应包含：
- 可能的故障原因（按可能性排序）
- 每个原因对应的检测方法（测什么、怎么测、正常值多少）
- 相关电路图和文档（附来源）
- 如果信息足够：给出维修建议
- 如果信息不足：告诉用户下一步该测什么，等反馈后继续分析

### System Prompt 中需要明确的约束

- 信息足够时立即生成回复，不要过度探索
- 当能同时获取多个维度的信息时，同时调用多个工具
- 连续向用户提问不超过 3 次
- 回复格式：结论在前、依据在后、附来源文档链接

## 三、一个完整的诊断示例

### 用户输入："EDC17CV44 报了 P01F5"

**Loop 第1轮：信息收集 + 多维度调查**

LLM 分析：有 ECU 型号 + 故障码，信息足够开始诊断。需要三个维度的信息，可以并行调用。

同时调用：
- dtc_diagnosis("P01F5", ecu="EDC17CV44") → 返回故障码含义、可能原因、相关传感器
- search_circuit_diagram("EDC17CV44 共轨压力传感器电路") → 返回相关电路图
- search_repair_knowledge("EDC17CV44 P01F5") → 返回类似案例的维修经验

**Loop 第2轮：综合研判 + 引导检测**

LLM 综合三个 tool 的返回结果，生成完整回复：

1. **故障解读**：P01F5 — 共轨压力过低，通常由共轨压力传感器信号异常、高压油泵供油不足或喷油器泄漏引起
2. **建议检测步骤**：
   - 第一步：测量共轨压力传感器供电电压（针脚X，正常 5V±0.1V）
   - 第二步：检查传感器信号线对地电阻
   - 第三步：如果传感器正常，检查高压油泵进油管路
3. **相关电路图**：附上 EDC17CV44 共轨系统电路图（标注了针脚位置）
4. **类似案例参考**：某维修案例中，同款 ECU 报 P01F5 最终是传感器插头腐蚀导致接触不良
5. **追问**："您方便测量一下共轨压力传感器的供电电压吗？正常应该是 5V±0.1V，如果偏差较大可以帮您进一步缩小故障范围。"

Loop 收束，等待用户反馈。

**后续交互：用户回答测量值**

用户："量了，只有 3.2V"

新一轮 Loop 启动：
- LLM 看到历史上下文（故障码 + 电路图 + 测量值 3.2V）
- 判断：供电电压偏低，可能是线路问题或 ECU 供电端问题
- 可能调 search_circuit_diagram 找供电线路的完整走向
- 给出进一步的排查建议："供电电压明显偏低，建议检查从 ECU 针脚X到传感器之间的线束，重点查接插件是否氧化、线路是否破损..."

**这就是 Agent Loop 带来的价值：不是一次性给个报告就结束，而是像老师傅一样一步步带着用户排查问题。**

## 四、模块串联关系

```
用户请求
    ↓
Pydantic AI Agent Loop
    │
    │  System Prompt（诊断思维框架）
    │       ↓
    │  LLM 推理 → 决定调哪些 tool（可并行多个）
    │       ↓
    │  ┌──────────────────────────────────────────┐
    │  │ Skill 层                                  │
    │  │ • search_circuit_diagram  → MCP Server 1  │
    │  │ • search_repair_knowledge → MCP Server 2  │
    │  │ • dtc_diagnosis           → 诊断服务       │
    │  │ • ask_user_question       → 中断等待用户   │
    │  └──────────────────────────────────────────┘
    │       ↓
    │  Tool 结果回灌到消息链
    │       ↓
    │  LLM 再次推理：还需要更多信息？
    │  • 是 → 继续 loop
    │  • 否 → 综合所有结果生成回复，退出 loop
    │
    ↓
SSE 流式返回前端
    │
    ├── tool_status 事件 → 前端显示"正在搜索电路图..."
    ├── text_delta 事件  → 流式渲染诊断结果
    ├── ask_user 事件    → 渲染问题卡片等待用户操作
    └── done 事件        → 本轮结束
```

串联的纽带是**消息链**：每次 tool 调用的输入和输出都追加到消息链中，LLM 每轮推理都能看到之前所有的 tool 结果和用户交互，基于完整上下文做下一步决策。

## 五、优化方向（按优先级排序）

### 1. 并行 tool 调用（效果最大）

上面例子中，dtc_diagnosis、search_circuit_diagram、search_repair_knowledge 三个调用之间没有依赖关系，可以在同一轮 loop 中并行发出。

4 轮串行变 2 轮并行，延迟直接砍半。

System Prompt 中引导 LLM："当你能明确需要多个维度的信息时，同时调用多个工具，不要串行。"

### 2. Tool 返回值质量

tool 返回的信息越精准越完整，LLM 越不需要二次查询。每少一轮 loop 就省 1-3 秒。

MCP Server 返回的不应该是原始检索结果的堆砌，而是经过整理的关键信息——提取要点、格式化、标注出处。这在 MCP Server 的业务适配层完成。

### 3. System Prompt 持续迭代

诊断思维框架写得越好，LLM 行为越合理——该问的问、该查的查、不该多跑的不多跑。

这不是一次性的工作，需要用真实用户问题反复测试：
- LLM 是不是选对了 tool？
- 是不是过度追问了？
- 是不是漏查了某个维度？
- 什么时候该收束但没收束？

每发现一个问题就调整 System Prompt 的措辞，这是长期最重要的优化。

### 4. 模型分层策略

全部走 loop 意味着 LLM 调用量更大，成本更敏感。

从便宜模型（DeepSeek V3）开始，观察哪些场景处理不好（tool 选错、参数填错、不该追问时追问），只在这些场景升级到更好的模型。

Pydantic AI 的 FallbackModel 支持自动降级：便宜模型先试，失败自动切换到贵模型。

### 5. 可观测性

全部走 loop 后，LLM 的决策路径不确定，必须能看到：
- 每轮 loop LLM 做了什么决策
- 调了什么 tool、传了什么参数
- tool 返回了什么
- 为什么收束 / 为什么没收束

这是所有后续优化的基础。看不到哪里有问题就无法优化。
