# 电路图内问答定位方案

## 1. 目标

在现有“图内关键词搜索”的基础上，增加一层“图内问答定位”能力。

用户不再只能输入一个关键词，而是可以直接输入自然语言问题，例如：

- `帮我找油门踏板到 ECU 的信号线`
- `EECU 供电保险是哪个`
- `尿素泵控制线在哪里`
- `起动继电器相关电路在什么位置`

系统需要根据问题自动拆解目标、扩展搜索词、调用现有图内搜索、排除无关命中，并最终返回：

- 最可能的位置
- 可直接打开 WebView 的 `page + points`
- 定位依据
- 少量备选候选

这不是做一个通用聊天机器人，而是做一个面向电路图内部定位的专用 agent 能力。

## 2. 为什么现在可以做第一版

当前项目已经具备以下基础能力：

- 已经可以用 `pdf_id + keyword` 调用外部图内搜索接口
- 图内搜索结果包含：
  - `page_index`
  - `bounding_box`
  - `context`
  - `reading_order`
  - `element_index`
- 已经有解析库映射能力：
  - `filename -> latest_pdf_id`
  - `latest_result_path`
  - `source_pdf_url`
- 已经可以把搜索命中的像素框转换成外部 WebView 所需的 `points`
- 全屏文档 WebView 已经支持：
  - `page`
  - `points`
  - 上一项 / 下一项切换
- 部分文档可以拿到 `latest_result_path` 对应的 OCR JSON

因此，现在已经具备做“基于多次搜索和上下文重排的图内问答定位”第一版的条件。

## 3. 当前能力边界

### 3.1 现在能做到什么

基于当前能力，系统可以：

- 从用户自然语言里抽取核心目标词
- 自动扩展若干同义词、缩写词、关联词
- 对同一文档执行多次图内搜索
- 合并多个关键词的候选命中
- 结合 `context`、`reading_order`、页码、空间邻近关系做重排
- 返回“最可能位置”和“备选位置”

### 3.2 现在做不到什么

基于当前能力，系统还做不到真正意义上的“完整图纸语义理解”：

- 不能稳定理解整页线路连接关系
- 不能直接理解线条拓扑、箭头跨页关系、元件图形语义
- 不能只靠一次推理就确定唯一正确答案
- 对 OCR 质量差、文字极少、图形主导的页面效果有限

所以第一版的定位本质是：

`多关键词召回 + OCR 上下文判断 + 候选重排`

而不是：

`完整电路图结构推理引擎`

## 4. 方案定位

本方案做的是：

`图内问答定位 agent`

不是：

- 通用聊天
- 通用电路图问答
- 自研电路图解析器
- 替换现有关键词搜索

关系应该是：

```text
现有图内关键词搜索 = 基础搜索工具
图内问答定位 agent = 搜索编排与定位决策层
WebView = 最终承载定位结果的查看器
```

## 5. 交互形态

建议继续使用现有全屏 WebView，不新建独立聊天页面。

在现有全屏 WebView 里，把顶部搜索胶囊升级为两种输入模式之一：

### 方案 A：直接升级为自然语言输入

输入框提示词改成：

`输入图内问题，例如“找油门踏板到 ECU 的信号线”`

优点：

- 用户理解成本低
- 前端改动最小
- 直接沿用现有搜索入口

缺点：

- 用户不容易区分“关键词搜索”和“智能定位”

### 方案 B：保留关键词搜索，再增加“图内助手”入口

例如：

- 左侧是普通搜索输入框
- 右侧增加一个 `图内助手` 按钮
- 点击后打开一个简化问答面板

优点：

- 语义更清晰
- 可以保留传统精确搜索入口

缺点：

- 前端交互复杂度更高

第一版建议优先做方案 A。

## 7. 推荐模块结构

建议新增：

```text
backend/app/agent/domain/circuit_body_search/
  query_agent.py
  query_normalizer.py
  query_planner.py
  candidate_ranker.py
```

职责划分：

### `query_normalizer.py`

负责把用户自然语言问题拆成结构化查询意图，例如：

```json
{
  "target_terms": [],
  "anchor_terms": [],
  "relation_terms": [],
  "original_message": ""
}
```

### `query_planner.py`

负责根据结构化意图生成多轮搜索任务，例如：

```text
搜索 1：油门踏板
搜索 2：加速踏板
搜索 3：APP
搜索 4：ECU
搜索 5：EECU
搜索 6：APP + ECU 关联判断
```

### `candidate_ranker.py`

负责把多轮搜索结果合并、去重、聚类和排序。

### `query_agent.py`

负责编排整条链路：

```text
接收问题
-> 解析意图
-> 生成搜索任务
-> 调现有 viewer/search 基础能力
-> 合并候选
-> 生成最佳定位点
```

## 8. 第一版执行链路

```text
用户输入自然语言问题
  ↓
解析目标对象 / 锚点对象 / 关系词
  ↓
生成 3~8 个搜索关键词
  ↓
逐个调用图内搜索接口
  ↓
收集 raw hits
  ↓
按页码和空间位置聚类
  ↓
结合 OCR 上下文做候选打分
  ↓
输出 best_hit + candidates
  ↓
前端 WebView 自动跳到 best_hit.page + best_hit.points
```

## 9. 查询理解策略

### 9.1 第一版不依赖 LLM 也能跑

为了保证稳定性，第一版可以先使用规则解析：

- 抽取主目标词
- 抽取关联对象词
- 抽取关系词
- 做有限同义词扩展

例如：

```text
油门踏板 -> 油门踏板 / 加速踏板 / APP / APS
ECU -> ECU / EECU
电源 -> 电源 / 供电 / 30 / 15 / 5V
搭铁 -> 地 / GND / 搭铁
```

### 9.2 第二版再引入 LLM

如果后续接入 LLM，建议只做：

- 查询归一化
- 候选重排解释

不要让 LLM直接替代原始召回，否则不稳定。

也就是：

```text
LLM 用来理解问题和排序
基础搜索仍然依赖现有图内搜索接口
```

## 10. 候选聚类与排序

第一版建议使用规则优先。

### 10.1 聚类

把多轮搜索命中按以下维度聚类：

- 同一页
- bbox 距离接近
- 上下文词相似
- reading_order 接近

聚类后每个 cluster 代表一个“候选区域”。

### 10.2 打分因子

建议打分项：

- 是否命中主目标词
- 是否命中锚点词
- 是否命中关系词
- 同一 cluster 内命中词种类数量
- 附近 OCR 文本是否出现更多相关词
- 是否位于标题页、目录页、说明页
- 是否只命中过于泛化的词，例如单独命中 `ECU`

### 10.3 降权规则

以下情况要明显降权：

- 只有泛词，没有主目标词
- 目录页
- 图例页
- 页面说明文字区
- 同页多个重复标签，但附近没有其他目标词支撑

## 11. 与现有接口的关系

### 11.1 现有 `/viewer/{token}/search`

保留，继续作为基础搜索能力。

职责：

- 关键词搜索
- 返回标准化命中
- 返回 `page + points`

### 11.2 新增 `/viewer/{token}/agent-locate`

新增，负责自然语言定位。

职责：

- 问题解析
- 多轮搜索编排
- 候选排序
- 返回最佳位置

不要把自然语言理解逻辑直接塞进 `/viewer/{token}/search`。

前端建议继续落在现有 [DocumentViewer.tsx](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/frontend/user/src/components/DocumentViewer.tsx)。

## 13. 为什么第一版不能直接做完整图内聊天

因为当前底层能力还缺少这些条件：

- 图形级结构化解析
- 线条连接关系
- 跨页导线关系追踪
- 元件图标识别
- 接插件与端子拓扑

所以现在适合做的是：

`问答驱动的图内定位`

而不是：

`完整图内知识问答系统`

如果现在直接承诺“像工程师一样理解整张图”，风险很高。

## 14. 第一版 MVP 范围

建议第一版只做以下能力：

- 输入自然语言问题
- 自动扩展有限搜索词
- 调用多次图内搜索
- 合并候选
- 返回一个最佳定位点
- 支持切换 2~5 个备选候选

第一版不做：

- 多轮对话记忆
- 跨页线路推理
- 图形元件识别
- 自动生成复杂维修解释
- 页面内完整聊天气泡流

## 15. 成功标准

第一版成功标准建议定义为：

- 对典型“找位置”问题，能比单关键词搜索更稳定地找到正确区域
- 能明显减少用户手动翻页和逐个试关键词的次数
- 返回结果可直接驱动 WebView 精确定位
- 错误时给出 2~5 个合理候选，而不是空结果或完全错误的唯一答案

## 16. 风险

主要风险：

- OCR 文本质量不稳定
- 命中词过于泛化，导致候选很多
- 同义词扩展不完整
- 没有 `latest_result_path` 的文档上下文不足
- 只靠文字仍然无法理解纯图形区域

对应策略：

- 第一版优先支持“文字标签明显”的图纸
- 保留候选切换，不强行只返回一个答案
- 把“定位依据”暴露给用户，便于快速判断结果对不对

## 17. 实施建议

建议分两步落地：

### 第一步

先做后端 `agent-locate` 接口和规则版候选排序。

目标：

- 能闭环跑通
- 能给出 `best_hit.points`

### 第二步

前端把现有搜索框升级成自然语言定位入口，并支持候选切换。

目标：

- 用户在现有 WebView 内直接使用
- 不新增重页面

## 18. 结论

基于当前项目现有能力，可以做一个可靠的第一版“图内问答定位”。

它的本质是：

`自然语言问题 -> 多关键词图内搜索 -> 候选聚类与排序 -> WebView 精确定位`

这条链路在当前架构下是成立的，而且和现有全屏 WebView、`page + points`、图内搜索接口天然兼容。

不建议第一版直接承诺完整图内聊天问答。

更合理的落地方向是先做：

`图内问答定位 agent`

等第一版验证效果后，再决定是否继续往更强的图纸语义理解推进。
