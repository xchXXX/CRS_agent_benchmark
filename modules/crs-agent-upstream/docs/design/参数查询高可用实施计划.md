# 参数查询高可用实施计划

## 1. 目标

将 `PARAM_QUERY` 作为第四个独立业务能力正式接入当前 Loop，并按高可用标准建设，不采用“先凑一个能跑的最小版本再返工”的策略。

本计划追求的不是“能回答几个示例问题”，而是：

- 外部参数知识库变化后能稳定增量同步
- 外部库偶发不可用时，当前系统仍能服务
- 查询结果建立在结构化证据之上，而不是 LLM 猜测
- 参数查询作为独立业务贯通后端、前端、反馈、生命周期与观测体系
- 后续扩展到 `dataflow`、`standard_datastream` 时不推翻现有结构

## 2. 非功能要求

## 2.1 可用性

- 外部参数库短时不可用时，系统继续使用本地最近一次成功同步的数据对外服务
- 启动同步失败不能拖垮主服务启动
- 单次同步异常不能污染当前可用索引

## 2.2 一致性

- 同一批同步结果必须以原子方式切换到运行时索引
- 结构化行必须与来源知识版本严格对应
- 解析器版本升级后支持全量可控重解析

## 2.3 性能

- 运行时查询优先命中内存索引，不依赖全文 Markdown 扫描
- 常规参数查询应尽量在单次数据库访问或零数据库访问内完成
- LLM 只处理小范围候选，不处理整库内容

## 2.4 可观测性

- 能看到同步状态、最近同步时间、失败原因、当前索引版本
- 能看到参数查询命中率、澄清率、失败率、外部库调用耗时

## 2.5 可运维性

- 支持启动自动同步
- 支持手动重同步
- 支持只重解析、不重拉取
- 支持参数能力独立开关

## 3. 总体技术路线

```text
外部 decoder_sit.ai_knowledge(pin_info)
    ↓ 只读拉取
本地镜像表 param_knowledge_source
    ↓ Markdown 解析
结构化行表 param_pin_row
    ↓ 别名补充
别名表 param_alias
    ↓ 内存索引构建
ParameterQueryIndexStore
    ↓
ParameterQueryService
    ↓
Loop Tool: query_parameters
    ↓
Runtime Adapter
    ↓
前端 PARAM_QUERY 业务展示
```

## 4. 高可用设计决策

## 4.1 外部库不是查询主路径

运行时参数查询不应直接查询外部 `decoder_sit`。

外部库只负责：

- 启动同步
- 定时同步
- 手动同步

用户请求主路径应依赖：

- 本地结构化表
- 内存索引

这样才能保证：

- 外部波动不影响在线查询
- 在线延迟稳定
- 查询行为可测可控

## 4.2 启动同步失败不阻断服务

启动时同步采用“尽力刷新”策略：

- 若同步成功，更新本地投影和索引
- 若同步失败，但本地已有可用快照，服务继续启动
- 若同步失败且本地没有任何快照，则 `PARAM_QUERY` 降级不可用，但其他业务正常

这意味着需要把参数查询能力与主服务生命周期隔离开。

## 4.3 索引切换必须原子化

同步与运行时索引不能共享可变对象。

正确做法：

1. 先在后台线程/任务中生成新索引快照
2. 完成后一次性替换内存引用
3. 在线查询始终只读取“当前有效索引”

不能边同步边改当前索引。

## 4.4 结构化行表允许覆盖重建

`param_pin_row` 属于派生数据，不是主业务数据。

因此允许：

- 按 `source_knowledge_id` 覆盖重建
- 根据 `parser_version` 全量重解析

这比做复杂 patch merge 更稳定。

## 4.5 运行时查询必须“先检索、后 LLM”

LLM 只用于：

- 小范围标准化
- 歧义消解
- 中文回答组织

不能让 LLM 负责：

- 直接扫全库
- 直接扫整篇 Markdown
- 直接生成 SQL

## 4.6 首期参数查询只接 `pin_info`

虽然业务名是 `PARAM_QUERY`，但首期只接：

- `pin_info`

原因：

- 数据结构稳定
- 场景边界清晰
- 结构化价值最高

未来再扩：

- `dataflow`
- `standard_datastream`

## 5. 系统级实施拆分

## 5.1 工作流 A：数据库与配置基建

### 目标

完成参数查询本地投影层和配置层建设。

### 任务

1. 执行新增表 DDL
   - `param_knowledge_source`
   - `param_pin_row`
   - `param_alias`
   - `param_sync_job`

2. 在配置中新增参数查询相关配置
   - 启用开关
   - 外部库连接
   - 连接超时
   - 同步批次大小
   - 解析器版本
   - 是否启动同步

3. 设计默认别名初始化逻辑
   - 字段级别系统别名
   - 常见口语映射

### 验收标准

- 本地表结构可用
- 配置可以完整加载
- 不配置参数查询时主服务不报错

## 5.2 工作流 B：外部仓储与同步引擎

### 目标

构建稳定的参数知识同步链路。

### 任务

1. 实现只读外部仓储 `ExternalParameterKnowledgeRepository`
   - 获取 `pin_info` 元信息列表
   - 获取单条正文
   - 严格超时控制
   - 标准错误映射

2. 实现 `ParameterKnowledgeSyncService`
   - 增量比对
   - 新增记录同步
   - 更新记录同步
   - 失效记录处理
   - 同步日志写入

3. 实现同步模式
   - 启动同步
   - 手动同步
   - 全量重解析

4. 同步过程使用“先写库、后切索引”的双阶段流程

### 高可用要求

- 同步失败不影响当前索引
- 单条记录解析失败不终止整批同步
- 同步日志可追踪失败明细
- 多实例场景下要预留分布式锁能力

### 验收标准

- 启动同步成功后本地投影完整
- 外部库不可用时不会拖死服务启动
- 重复执行同步结果幂等

## 5.3 工作流 C：Markdown 解析器

### 目标

将 `pin_info` Markdown 稳定解析为结构化针脚行。

### 任务

1. 解析文档头部说明段
2. 识别主表格
3. 标准化列名
4. 清洗单元格空白与符号
5. 解析电压区间
6. 保留原始 `raw_row_json`

### 高可用要求

- 对列顺序变化具备容错
- 对中英文括号、全角半角差异具备容错
- 对个别缺列、空值、格式不整齐具备容错
- 无法解析时明确落 `parse_status=failed`

### 验收标准

- 现有 `pin_info` 主流样本可稳定解析
- 行数与表格内容一致
- 电压文本与原文可追溯

## 5.4 工作流 D：归一化与候选召回

### 目标

将用户口语化问题转为标准查询对象。

### 任务

1. 规则提取器
   - 提取针脚号
   - 提取电压制式
   - 提取显式字段词

2. 别名服务
   - 字段别名
   - ECU 别名
   - 零部件别名

3. 候选召回
   - ECU 候选
   - 文档候选
   - 部件候选
   - 字段候选

4. 小范围 LLM 规范化器
   - 输入：用户问题 + 候选集合
   - 输出：标准查询对象

### 高可用要求

- 规则可先行命中，LLM 只是补充
- LLM 失败时能退回到规则+候选的保守模式
- 候选数量必须受控，不能把整库内容喂给模型

### 验收标准

- 常见口语问法能映射到标准 ECU / 部件 / 字段
- 未知 ECU 或未知部件能明确识别为需澄清或无结果

## 5.5 工作流 E：运行时参数查询服务

### 目标

构建面向 Loop 的统一参数查询服务。

### 任务

1. 构建 `ParameterQueryService`
2. 实现 `query(query, selection_payload)` 主入口
3. 支持三类输出
   - `ok`
   - `need_clarify`
   - `failed`

4. 输出 evidence package

### 高可用要求

- 查询优先命中内存索引
- 如果内存索引不可用，可退回本地数据库查询
- 如果本地投影缺失，明确返回不可用状态，而不是胡答

### 验收标准

- 命中准确
- 澄清逻辑单轮可收敛
- 返回结构统一稳定

## 5.6 工作流 F：Loop 接入

### 目标

将参数查询接入 Agent Loop 成为第四独立业务。

### 任务

1. `IntentRouter` 新增 `PARAM_QUERY`
2. `AgentFactory` 新增 `query_parameters` tool
3. 更新系统提示词
4. 在 runtime 中新增：
   - `PARAM_QUERY` business 映射
   - `query_parameters` tool 截获与响应组装
   - 参数查询 ask_user 恢复逻辑

### 高可用要求

- 参数查询不能误路由到资料搜索
- 参数查询 ask_user 恢复必须可重放
- 参数查询失败不能破坏其他业务状态

### 验收标准

- `PARAM_QUERY` 在 Loop 中独立运转
- 多轮交互能稳定恢复

## 5.7 工作流 G：前端业务接入

### 目标

让 `PARAM_QUERY` 成为用户可感知的第四业务。

### 任务

1. 类型定义新增 `PARAM_QUERY`
2. 业务切换展示新增 `参数查询`
3. 示例问题新增参数类问题
4. `param_request` 真正接入消息渲染
5. 反馈标签新增 `PARAM_QUERY`

### 高可用要求

- 新业务不影响现有三业务展示
- 参数查询结果卡片结构清晰、手机端优先
- 澄清卡片可复用已有 ask_user 交互

### 验收标准

- 参数查询在 UI 中有独立业务标识
- 查询结果、澄清、反馈链路完整

## 5.8 工作流 H：观测性与运维

### 目标

建立可观测、可运营的参数查询能力。

### 任务

1. tracer 埋点
   - 同步开始/结束
   - 外部库调用耗时
   - 解析成功/失败
   - 查询命中类型
   - 澄清触发原因

2. 健康状态扩展
   - 参数查询是否启用
   - 是否已有可用索引
   - 最近同步时间
   - 最近同步状态

3. 管理端或接口侧预留状态查看能力

### 高可用要求

- 出问题时能快速区分是：
  - 外部库不可用
  - 同步失败
  - 解析失败
  - 运行时归一化失败
  - 前端渲染问题

### 验收标准

- 同步链路与查询链路都可观测

## 6. 关键运行策略

## 6.1 双层回退

运行时查询优先级：

1. 当前有效内存索引
2. 本地结构化表
3. 明确降级失败

不允许直接回退到：

- 外部实时全文查询
- LLM 直接读整篇 Markdown

## 6.2 启动同步策略

建议：

- 服务启动时触发参数同步
- 同步失败时不阻断服务启动
- 若本地无索引则 `PARAM_QUERY` 标记不可用

## 6.3 手动重同步策略

后续应补一个管理接口：

- 触发参数同步
- 触发全量重解析
- 查看最近任务状态

## 6.4 索引刷新策略

索引刷新不做增量打补丁，采用：

- 从当前数据库重新装载
- 生成新快照
- 原子替换

这样最稳。

## 7.1 单元测试

- 电压范围解析
- 标题 ECU 提取
- 别名匹配
- 规则提取

## 7.3 回归测试

- 资料搜索
- 故障诊断

## 8. 实施顺序

以下顺序不是 MVP，而是按系统依赖关系拆分的正式实施顺序。

### 第一阶段：基础设施

1. 表结构 DDL
2. 配置项
3. 域模型

### 第二阶段：同步与解析

1. 外部仓储
2. 同步服务
3. Markdown 解析器
4. 同步日志

### 第三阶段：查询核心

1. 别名服务
2. 内存索引
3. 归一化器
4. 参数查询主服务

### 第四阶段：Loop 接入

1. Intent
2. Tool
3. Runtime 响应适配
4. ask_user 恢复

### 第五阶段：前端与观测

1. Business 类型接入
2. param_request 渲染
3. 反馈与切换
4. 健康状态与埋点

### 第六阶段：全量测试与上线准备

1. 测试矩阵执行
2. 同步校验
3. 实际样本问句回归
4. 灰度开关验证

## 9. 上线准备要求

上线前至少完成：

1. 外部 `pin_info` 全量同步一次
2. 典型 ECU 样本抽检
3. 常见口语问句回归
4. 参数查询能力开关验证
5. 启动同步失败回退验证

## 10. 结论

高可用版本的参数查询不是“多加一个 skill”，而是一套完整能力：

- 外部知识同步
- 结构化解析
- 本地投影
- 内存索引
- 口语归一化
- Loop 独立业务接入
- 前端业务接入
- 可观测与可回退

后续正式开发应严格按本计划推进，而不是先做一个临时版再返工。
