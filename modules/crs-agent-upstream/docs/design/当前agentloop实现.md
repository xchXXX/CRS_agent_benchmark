# 当前 AgentLoop 实现

## 1. 文档目的

这份文档只描述当前 AgentLoop 的业务结构、状态流转和循环方式，不讨论代码细节。

当前 AgentLoop 的职责不是“单次调用大模型然后直接返回”，而是作为一个统一编排层，持续管理下面几件事：

1. 判断这次请求属于哪条业务线。
2. 决定是延续原会话，还是切断旧上下文重新开始。
3. 驱动一次请求内部的模型与工具协作。
4. 在需要用户补充信息时暂停，并把暂停点保存下来。
5. 在用户下一次补充后，从暂停点恢复并继续推进。

所以它本质上是一个“可暂停、可恢复、跨多轮对话持续推进”的运行循环。

## 2. AgentLoop 的核心结构

当前实现可以从五个结构来看：

| 结构层 | 作用 |
| --- | --- |
| 请求入口层 | 接收普通问答和流式问答请求，并为本次请求准备身份、token、业务依赖 |
| 意图判定层 | 把请求分到文档搜索、故障诊断、普通问答等业务线 |
| 主执行层 | 负责把用户输入、历史上下文、待恢复状态交给主 Agent 继续运行 |
| 状态存储层 | 保存会话历史、待恢复的 ask user 状态、流式中的临时状态 |
| 结果整形层 | 把运行结果统一整理成前端能理解的 `message`、`documents`、`ask_user`、`error` 等响应 |

其中真正让 AgentLoop 形成“循环”的，不是一次请求内的连续调用，而是“状态存储层 + 下一次恢复”的配合。

## 3. 两层 Loop

当前实现实际上有两层 loop。

### 3.1 第一层：单次请求内循环

这是一次请求进入后，在后端内部发生的推进过程。

它的典型节奏是：

1. 收到请求。
2. 恢复历史上下文或待恢复状态。
3. 判断业务线。
4. 把当前输入交给主 Agent。
5. 主 Agent 在这一轮内部与工具多次来回。
6. 得到一个阶段性结果。
7. 将阶段性结果整理成前端响应。

这一层 loop 的关键点是：

- 它不是“用户一句，模型一句”的简单往返。
- 在一次请求内部，主 Agent 可能已经完成了多轮工具调用和决策。
- 这一层什么时候结束，不取决于模型有没有说完，而取决于是否到达了一个可对前端返回的状态。

可返回状态主要有四类：

| 状态 | 含义 |
| --- | --- |
| `message` | 返回普通文本答复 |
| `documents` | 返回结构化文档结果 |
| `ask_user` | 当前请求暂停，等待用户补充信息后再继续 |
| `error` | 当前请求失败，无法继续推进 |

### 3.2 第二层：跨请求外循环

这是当前 AgentLoop 最核心的部分。

它的节奏是：

1. 本轮请求推进到某个暂停点。
2. 返回 `ask_user` 给前端。
3. 前端把用户补充的信息再次发回来。
4. AgentLoop 根据 `session_id` 和 `tool_call_id` 找回上一次暂停时保存的状态。
5. 从暂停点继续推进，而不是从头开始。
6. 如果还不够明确，就再次暂停。
7. 反复循环，直到产出最终结果。

所以当前 AgentLoop 的“loop”不是单纯的内部死循环，而是：

`运行一段 -> 暂停 -> 保存状态 -> 等用户补充 -> 恢复 -> 再运行一段`

这也是它能支持澄清、补充信息、继续搜索、继续诊断的根本原因。

## 4. 进入 Loop 之前的准备动作

每次请求进入时，AgentLoop 先做四类准备。

### 4.1 生成请求作用域

这一层会为当前请求分配本次请求标识，并拿到当前用户身份、token、业务依赖。

这里的重点不是“鉴权页面”本身，而是：

- 当前请求使用的是请求级依赖。
- 外部文档搜索能力依赖有效 token。
- 同一个 AgentLoop 在不同请求中，可以拿到不同的用户身份和 token。

### 4.2 校验运行时是否可用

如果主运行时不可用，这一轮不会进入主 loop，而是直接返回错误。

### 4.3 判断是否需要直接恢复文档搜索澄清

这是当前实现里一个非常重要的特殊分支。

如果本次请求本身就是在回答上一次文档搜索的澄清问题，那么系统不会再把这次输入重新丢给主 Agent 从头思考，而是直接：

1. 找回上一次保存的文档搜索澄清状态。
2. 读取用户这次选中的筛选条件。
3. 直接重新执行文档搜索。
4. 继续做歧义分析。
5. 决定是继续澄清，还是直接返回文档结果。

这意味着文档搜索的多轮澄清，当前实现中走的是“业务闭环恢复”，而不是“再次让大模型自由发挥”。

### 4.4 恢复历史或清理历史

当前实现会根据请求内容，进入三种上下文模式之一：

| 模式 | 说明 |
| --- | --- |
| 延续历史 | 读取原会话历史，继续当前话题 |
| 恢复暂停点 | 读取之前保存的 ask user 状态，把用户补充拼回上一轮上下文里 |
| 清空历史 | 当用户明确确认切换话题时，先丢掉旧上下文，再开始新一轮 |

这里的“清空历史”是当前生命周期控制的核心动作。也就是说，AgentLoop 真正依赖的不是前端显示什么标签，而是用户是否明确确认切换话题。

## 5. 意图分流

准备完成后，AgentLoop 会先判断当前请求应该进入哪条业务线。

当前主业务线有三条：

| 业务线 | 目标 |
| --- | --- |
| 文档搜索 | 查资料、手册、电路图、图纸、说明书等 |
| 故障诊断 | 带故障码或进入诊断模式的请求 |
| 普通问答 | 一般技术咨询、维修经验、解释说明类问题 |

这一步的意义非常大，因为后续 loop 的推进方式不同：

- 文档搜索偏结构化，强调检索、筛选、澄清和结构化结果。
- 故障诊断偏流程型，强调故障码识别、ECU 澄清和诊断结果。
- 普通问答偏知识型，强调直接回答或补充维修知识上下文后回答。

## 6. 单次请求内的主推进过程

一旦进入主执行阶段，本轮请求就进入“主 Agent 推进期”。

当前可以把它理解成下面的状态流：

| 阶段 | 发生的事 | 可能去向 |
| --- | --- | --- |
| 运行开始 | 带着当前输入、会话历史、恢复结果进入主 Agent | 工具协作 / 直接输出 |
| 工具协作 | 主 Agent 根据业务需要调用搜索、诊断、维修知识等工具 | 继续协作 / 形成结果 |
| 阶段产出 | 主 Agent 形成文本输出，或提出 ask user，或产出工具结果 | 结果整形 |
| 结果整形 | 系统根据业务规则把结果转成统一响应 | `message` / `documents` / `ask_user` / `error` |

这里最重要的一点是：

AgentLoop 最终返回给前端的内容，不一定等于模型最后一句自然语言。

在当前实现里，系统会根据工具结果重新解释最终响应形态，尤其是文档搜索业务线。

## 7. ask user 在当前实现中的角色

ask user 不是一个普通提示框，而是 AgentLoop 实现跨轮 loop 的关键机制。

当前 ask user 有两种来源。

### 7.1 第一类：主 Agent 主动发起的 ask user

这种场景下，主 Agent 会认为当前信息不够，需要用户补充。

这时 AgentLoop 会做四件事：

1. 把这次暂停时的完整上下文保存下来。
2. 记录这次暂停对应的 `tool_call_id`。
3. 把问题、选项、补充方式整理成 `ask_user` 响应。
4. 返回前端等待用户回答。

当用户回答后，AgentLoop 会：

1. 用 `session_id + tool_call_id` 找到旧状态。
2. 把用户答案作为“上一个待完成动作的结果”接回去。
3. 重新进入主 Agent。
4. 让主 Agent 从原来的停点继续往后推。

这是一种“通用恢复式 loop”。

### 7.2 第二类：文档搜索后处理生成的 ask user

文档搜索比较特殊。

当前实现不是简单地让模型看到搜索结果后自由组织下一句，而是在搜索结束后再做一层结构化判断：

1. 结果是否有效。
2. 结果是否过多。
3. 是否存在品牌、车系等维度冲突。
4. 是否需要进一步澄清。

如果需要澄清，系统会直接生成一个结构化 ask user，并保存下一轮继续搜索所需的信息。

这种情况下，下一轮恢复时往往不需要再回到主 Agent，而是直接：

1. 读取原查询。
2. 合并用户选中的筛选条件。
3. 再次搜索。
4. 再次分析歧义。
5. 继续澄清或结束。

这是一种“业务直连式 loop”。

### 7.3 两类 ask user 的本质区别

| 类型 | 谁决定提问 | 恢复后先回到哪里 |
| --- | --- | --- |
| 通用 ask user | 主 Agent | 先回到主 Agent |
| 文档搜索 ask user | 文档搜索后处理层 | 先回到文档搜索业务闭环 |

这一点是当前 AgentLoop 最需要单独理解的地方。

## 8. 文档搜索这条 Loop 是怎么转的

文档搜索的 loop 当前是最完整的，也是结构最清晰的。

它的实际节奏如下：

1. 用户输入资料类需求。
2. 系统把请求判定为文档搜索。
3. 主 Agent 先进入搜索链路。
4. 搜索结果返回后，系统进入结构化后处理。
5. 后处理判断结果是否足够明确。
6. 如果不明确，返回结构化澄清问题。
7. 用户选择品牌、系列、车型等条件。
8. 系统根据用户选择重新搜索。
9. 再次做歧义分析。
10. 如果仍不明确，继续 ask user。
11. 如果已经明确，返回结构化文档结果。
12. 如果结果无效，则返回结构化无结果提示。

所以文档搜索不是“搜一次就结束”，而是：

`搜索 -> 分析 -> 澄清 -> 再搜索 -> 再分析 -> 直到收敛`

这就是当前文档搜索 loop 的核心。

## 9. 故障诊断这条 Loop 是怎么转的

故障诊断链路和文档搜索不同，它的核心不是结果筛选，而是诊断推进。

当前实现可以概括为：

1. 用户输入故障码或进入诊断模式。
2. 系统判断是否走外部诊断能力。
3. 如果诊断能力可用，则进入诊断业务线。
4. 如果某个故障码对应多个 ECU，系统会暂停并 ask user。
5. 用户选定 ECU 后，系统从暂停点恢复。
6. 继续推进诊断。
7. 最终返回报告、链接或诊断说明。

如果诊断服务不可用，则会退化成普通大模型解释型回答，不再走外部诊断闭环。

所以故障诊断 loop 的重点是：

`识别故障码 -> 需要时澄清 ECU -> 继续诊断 -> 输出结果`

## 10. 普通问答这条 Loop 是怎么转的

普通问答链路相对简单，但仍然是 loop 结构的一部分。

它的节奏通常是：

1. 请求被判断为普通问答。
2. 主 Agent 直接回答，或先补充维修知识上下文再回答。
3. 如果信息不足，也可能 ask user。
4. 用户补充后，系统从暂停点恢复并继续回答。

这条线更多是“知识补充式 loop”，没有文档搜索那种强结构化收敛过程。

## 11. 流式与非流式的关系

当前实现里，流式和非流式共用同一套核心状态机，只是对外暴露方式不同。

### 11.1 非流式

非流式的特点是：

- 后端内部推进完整轮次后，再一次性返回最终响应。
- 前端只看到最终态，不看到中间推进过程。

### 11.2 流式

流式的特点是：

- 请求开始时先进入开始态和处理中态。
- 如果业务允许暴露中间文本，则持续推送文本增量。
- 完成后再推送最终态。

但是当前实现里，文档搜索是一个特殊业务：

- 文档搜索的最终结果常常不是自然语言，而是 `documents` 或 `ask_user`。
- 因此当前实现会压住文档搜索过程中的临时文本，不让前端先显示一段大模型口语，再被结构化结果覆盖。

所以流式并没有改变 AgentLoop 的状态结构，只是改变了状态对外发送的时机。

## 12. 中断与恢复

流式中断时，当前实现不会简单丢弃这次上下文，而是尽量把已产生的部分内容保存进会话历史。

这样做的意义是：

1. 用户主动中断后，不会让会话上下文完全断裂。
2. 下一轮继续时，系统仍能感知到之前已经输出过什么。

这说明当前 AgentLoop 不只是管理“完整完成的请求”，也在管理“被中断但需要保留上下文”的请求。

## 13. 当前实现中的关键状态

如果只看状态，不看实现细节，可以把当前 AgentLoop 概括成下面这些核心状态：

| 状态 | 说明 |
| --- | --- |
| 新请求进入 | 本轮请求开始 |
| 运行时就绪检查 | 判断系统是否可运行 |
| 历史恢复 / 清理 | 决定延续旧话题还是重开新话题 |
| 意图分流 | 决定进入哪条业务线 |
| 主执行中 | 当前请求正在由主 Agent 和工具推进 |
| 文档搜索后处理 | 对搜索结果做有效性和歧义判断 |
| 等待用户补充 | 当前轮暂停，等待 ask user 回答 |
| 业务直连恢复 | 文档搜索澄清后不经主 Agent，直接继续业务闭环 |
| 通用恢复 | 把用户答案接回主 Agent，继续推进 |
| 终态输出 | 返回 `message`、`documents` 或 `error` |
| 流式中断保存 | 把中断内容记回历史，等待后续继续 |

## 14. 当前 AgentLoop 的实现特点

结合现在的结构，可以把它总结成下面几个特点。

### 14.1 它是“有状态”的

核心不是单次模型调用，而是：

- 会话历史会被保存。
- ask user 暂停点会被保存。
- 流式中断的部分内容也会尽量保存。

### 14.2 它是“按业务线分流”的

不同业务不是共用同一套输出逻辑，而是：

- 文档搜索偏结构化闭环。
- 故障诊断偏流程推进。
- 普通问答偏文本回答。

### 14.3 它是“后处理强于模型口头表达”的

尤其在文档搜索业务里，最终返回什么，不由模型最后一句话决定，而由业务后处理决定。

### 14.4 它是“分段式 loop”

不是一次调用从头跑到尾，而是：

`推进一段 -> 存状态 -> 等用户 -> 再恢复 -> 再推进`

## 15. 一句话总结

当前 AgentLoop 可以理解为一个统一的对话业务编排器：

它先分流业务，再在单次请求内部推动模型和工具协作；一旦遇到需要补充信息的节点，就暂停并保存状态；等用户补充后，再从暂停点恢复，继续推进，直到收敛为文本答复、文档结果或诊断结果。

如果只看“loop”本身，当前实现的本质就是：

`请求进入 -> 推进 -> 产生暂停点或结果 -> 保存状态 -> 下一轮恢复 -> 继续推进 -> 直到终态`
