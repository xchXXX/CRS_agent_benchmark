# 参数查询接入落地清单

## 1. 目标

将“参数查询”接入当前 Loop，作为第四个独立业务能力：

- `DOC_SEARCH`
- `FAULT_DIAGNOSIS`
- `GENERAL_CHAT`
- `PARAM_QUERY`

其中 `PARAM_QUERY` 首期只消费外部知识库中的 `pin_info` 数据，并通过本地结构化投影层提供高精度查询。

## 2. 首期能力边界

首期支持的问题类型：

- 查某个 ECU 的某个针脚作用
- 查某个 ECU 的某个零部件针脚电压
- 查某个 ECU 某类信号在几号脚
- 查某个针脚的开路电压 / 静态电压 / 低怠速电压

首期不做：

- 波形类分析
- 数据流区间分析
- 多文档聚合比对
- 跨 `pin_info` 与 `dataflow` 的联合推理

## 3. 建议的后端模块划分

建议新增目录：

```text
backend/app/agent/domain/parameter_query/
  __init__.py
  models.py
  parser.py
  alias_service.py
  index_store.py
  external_repository.py
  sync_service.py
  normalizer.py
  service.py
  response_adapter.py
```

### 3.1 `models.py`

定义核心对象：

- `ParameterQueryIntent`
- `ParameterQueryObject`
- `ParameterEvidencePackage`
- `ParameterClarifyOption`
- `ParameterQueryResult`
- `ParameterSyncStats`

### 3.2 `parser.py`

负责将 `pin_info` Markdown 解析成结构化表格行：

- 识别 Markdown 表格
- 标准化列名
- 解析单元格
- 解析电压范围
- 输出 `param_pin_row` 可落库对象

### 3.3 `alias_service.py`

负责：

- 维护字段级别系统别名
- 生成 ECU / 零部件的自动别名
- 从 `param_alias` 读取别名映射

### 3.4 `index_store.py`

负责运行时内存索引：

- `docs_by_ecu_name`
- `docs_by_voltage`
- `rows_by_doc_pin`
- `rows_by_doc_component`
- `rows_by_doc_definition`
- `field_alias_map`
- `ecu_alias_map`
- `component_alias_map`

同步完成后由这里统一重建。

### 3.5 `external_repository.py`

负责只读访问外部 `decoder_sit`：

- 查询 `ai_knowledge_type`
- 查询 `pin_info` 元信息
- 查询单条知识正文

注意：

- 默认只读
- 不承担业务归一化
- 不写入外部表

### 3.6 `sync_service.py`

负责：

- 启动同步
- 增量比对
- 拉取变更正文
- 重解析
- 覆盖更新本地表
- 写入 `param_sync_job`
- 触发索引重建

### 3.7 `normalizer.py`

负责把口语化问题转换为标准查询对象：

1. 规则提取
2. 别名归一化
3. ECU / 零部件候选召回
4. 小范围 LLM 规范化

它输出的不是自然语言，而是：

```json
{
  "intent": "PARAM_QUERY",
  "ecu_name": "EDC17C53",
  "system_voltage": 12,
  "pin_no": "K46",
  "component_name": "点火开关T15",
  "requested_fields": ["pin_definition"]
}
```

### 3.8 `service.py`

参数查询主服务，对外暴露统一调用入口：

- `sync_on_startup()`
- `refresh_indexes()`
- `query(query: str, selection_payload: dict | None = None) -> dict`

主流程：

1. 构建或恢复标准查询对象
2. 命中文档
3. 命中行
4. 生成 evidence package
5. 如有歧义，返回 `need_clarify`
6. 如命中明确，返回结构化结果

### 3.9 `response_adapter.py`

作用类似现有 `DocSearchResponseAdapter`。

负责：

- 从 tool envelope 中抽取参数查询结果
- 构建 `ask_user_question`
- 构建 `param_request` 响应内容
- 维护 `selection_payload` 恢复逻辑

## 6.4 工具设计

首期建议只新增一个主工具：

- `query_parameters`

不建议拆成多个 Tool 给模型自己编排，因为参数查询的歧义判断、候选召回、结构化检索本身更适合由服务层统一完成。

### Tool 签名建议

```python
query_parameters(
    query: str,
    selection_payload: dict | None = None,
) -> dict
```

### Tool 返回协议建议

统一 envelope：

```json
{
  "status": "ok | need_clarify | failed",
  "data": {...},
  "clarify": {...}
}
```

其中：

- `ok`：返回 evidence package
- `need_clarify`：返回候选项和上下文
- `failed`：返回错误信息

这样运行时只需要仿照 `doc_search` 做一个参数查询响应适配器。

## 7. 推荐的运行时处理方式

参数查询不建议完全依赖模型自己处理 tool return，而建议在 runtime 特判。

建议新增类似逻辑：

- `_maybe_build_parameter_query_response(...)`
- `_finalize_parameter_query_response(...)`

处理方式参考 `doc_search`：

1. 截获最新 `query_parameters` tool envelope
2. 如果是 `need_clarify`，构造 `ask_user`
3. 如果是 `ok`，返回 `ChatResponse(type="param_request", business="PARAM_QUERY")`
4. 如果是失败，给出标准错误响应

## 9. `selection_payload` 设计

参数查询需要的澄清 payload 应尽量小：

```json
{
  "ecu_name": "EDC17C53",
  "system_voltage": 12,
  "component_name": "点火开关T15",
  "requested_fields": ["pin_definition"]
}
```

如果只是选择某个 ECU 候选，则只传：

```json
{
  "ecu_name": "EDC17C53",
  "system_voltage": 12
}
```

恢复时仍然走：

- 原始 query
- selection_payload

然后再次调用 `query_parameters`。

## 10. 推荐的本地查询顺序

### 10.1 第一优先：规则命中

优先识别：

- 针脚编号
- 电压字段
- 明显的 ECU 关键词

### 10.2 第二优先：别名映射

对口语词做标准化：

- `作用/定义/功能`
- `开路电压/空载电压`
- `怠速电压`

### 10.3 第三优先：候选召回

从内存索引中拿到：

- ECU 候选
- 零部件候选
- 字段候选

### 10.4 第四优先：小范围 LLM 规范化

LLM 只在候选集中选择：

- 哪个 ECU
- 哪个部件
- 要查哪些字段

### 10.5 第五优先：结构化行命中

基于 `param_pin_row` 精确命中。

## 12. 最终建议

参数查询能力的首期实现，最合理的技术路线是：

1. 启动同步 `pin_info`
2. 解析为本地结构化行
3. 构建内存索引
4. 将口语输入规范化为标准查询对象
5. 用统一 `query_parameters` 工具接入 Loop
6. 用独立业务 `PARAM_QUERY` 接前后端生命周期

不要让模型直接“记住所有标题”，也不要让模型直接扫外部 Markdown。  
模型只负责在小候选集中做规范化和表达，真正的命中逻辑必须掌握在参数查询服务里。
