# 参数查询功能设计方案

## 1. 背景与目标

当前系统已有三个业务能力：

- 资料搜索
- 故障诊断
- 维修问答

现在需要新增第四个独立业务能力：`参数查询`。

本能力的典型问题不是“帮我找文档”，也不是“帮我诊断故障”，而是：

- `EDC17 的 K46 是什么作用？`
- `易控 F17 油门踏板 1 的开路电压是多少？`
- `这台 ECU 的点火开关针脚电压正常多少？`
- `MD1CE100 的 CANH 在哪个针脚？`

这类问题有几个明显特点：

1. 用户表达非常口语化，不会直接说出数据库中的完整标题，例如 `EDC17C53针脚电压(12V系统)`。
2. 用户可能只说 ECU 的简称、部件俗称、针脚号的一部分，甚至只说“那个点火开关线”。
3. 查询目标是强结构化字段，不适合把整篇 Markdown 文档直接交给 LLM 现读现答。
4. 外部数据源会持续变化，因此本地结构化数据必须支持增量同步和重解析。

本设计目标是：

- 把 `参数查询` 作为独立第四业务接入 Loop
- 让系统能把口语化用户输入规范化成数据库可执行的查询
- 让最终查询建立在结构化行数据之上，而不是建立在全文 Markdown 猜测之上
- 通过本地投影层承接外部知识库变化

## 2. 外部数据源现状

外部只读数据源：

- MySQL 库：`decoder_sit`
- 主表：`ai_knowledge`
- 类型表：`ai_knowledge_type`

当前与参数查询直接相关的数据类型：

- `pin_info`：针脚信息

现状确认：

- `pin_info` 类型已有有效数据 111 条
- 内容格式大多为 `md`
- 内容主体为较稳定的 Markdown 表格

常见列：

- `零部件`
- `ECU针脚编号`
- `针脚定义`
- `接插件针脚号`
- `开路电压（V）`
- `连接线束后静态电压`
- `低怠速电压`
- `备注`

这说明 `pin_info` 非常适合先解析为结构化行，再做查询。

## 3. 设计原则

### 3.1 外部库只做事实源

外部 `decoder_sit.ai_knowledge` 继续作为唯一事实源。

- 不修改
- 不删除
- 默认只读

当前系统只负责：

- 读取外部记录
- 同步元信息和正文
- 在本地构建可查询投影

### 3.2 本地库做结构化投影

当前系统本地库 `crs_agent` 新增参数查询专用表，用于：

- 保存外部知识的本地镜像元数据
- 保存解析后的结构化针脚行
- 保存别名映射
- 保存同步日志

本地表不是主数据，只是：

- 可重建
- 可覆盖
- 可重解析

### 3.3 AI 不直接查数据库

LLM 不直接执行 SQL，也不直接读取完整 Markdown 全文。

正确分工应为：

- 后端参数查询服务：负责候选召回、字段归一化、结构化检索
- LLM：负责理解口语问题、在小范围候选中做规范化选择、组织中文答案

### 3.4 先确定性检索，再 LLM 表达

参数查询是高精度场景，优先级必须是：

1. 规则提取
2. 别名归一化
3. 候选集召回
4. 小范围 LLM 规范化
5. 结构化证据生成
6. LLM 输出答案

不能让 LLM 直接“看一篇文档猜结果”。

## 4. 整体架构

```text
用户问题
  ↓
Intent Router 识别为 PARAM_QUERY
  ↓
参数查询归一化层
  - 提取 pin / ECU / 零部件 / 参数字段 / 电压制式
  - 别名映射
  - 候选召回
  - 小范围 LLM 规范化
  ↓
参数查询服务
  - 命中文档
  - 命中结构化针脚行
  - 生成 evidence package
  ↓
LLM 组织最终回答
  ↓
前端展示 PARAM_QUERY 业务消息
```

## 5. 为什么不能只注册一个 skill

当前项目里的独立业务不是简单“注册一个 skill 名称”就完成的，而是由以下几层共同构成：

1. `IntentRouter` 路由
2. Agent 工具注册
3. 系统提示词中的业务规则
4. Runtime `business` 映射
5. 前端业务类型与生命周期

因此 `参数查询` 应作为独立第四业务接入，业务名建议为：

- `PARAM_QUERY`

## 6. 查询流程设计

## 6.1 启动同步流程

系统启动时执行参数知识同步：

1. 读取外部 `ai_knowledge_type`，确认 `pin_info` 对应的 `type_id`
2. 只查询外部 `pin_info` 的元信息：
   - `id`
   - `type_id`
   - `title`
   - `content_format`
   - `status`
   - `is_deleted`
   - `latest_version`
   - `published_version`
   - `update_time`
3. 与本地 `param_knowledge_source` 对比
4. 对于新增或变更记录，再读取全文 `content`
5. 解析 Markdown 表格为结构化行
6. 覆盖更新本地 `param_pin_row`
7. 重建内存索引

需要重解析的条件：

- 本地不存在该 `source_knowledge_id`
- 外部 `update_time` 变化
- 外部 `latest_version` 变化
- 本地 `content_hash` 与新内容不一致
- 本地 `parser_version` 不是当前版本
- 本地 `parse_status != ready`

## 6.2 用户请求处理流程

### 第一步：意图识别

将以下表达判定为 `PARAM_QUERY`：

- 查某个 ECU 的针脚
- 查某个针脚定义
- 查某个部件针脚对应的电压
- 查某个 ECU 的 CANH/CANL/点火开关/油门踏板/主继电器等参数

典型关键词：

- `针脚`
- `定义`
- `电压`
- `开路电压`
- `静态电压`
- `低怠速电压`
- `哪个针脚`
- `几号脚`
- `CANH`
- `CANL`

### 第二步：规则提取

先用规则从用户问题中提取显式槽位：

- `pin_no`
  - 例如：`K46`、`A-17`、`B-86`、`03`
- `system_voltage`
  - 例如：`12V`、`24V`
- `field`
  - 例如：`针脚作用`、`定义`、`开路电压`、`静态电压`、`低怠速电压`

### 第三步：候选召回

基于本地索引召回候选：

- ECU 候选
- 文档候选
- 零部件候选
- 字段候选

注意，这一步不是让 LLM 看全库，而是先缩小范围。

### 第四步：小范围规范化

将以下内容交给 LLM：

- 用户原始问题
- 规则提取结果
- TopN ECU 候选
- TopN 零部件候选
- 字段别名映射结果

让 LLM 输出标准化查询对象，而不是自然语言回答。

标准化查询对象示例：

```json
{
  "intent": "PARAM_QUERY",
  "ecu_name": "EDC17C53",
  "system_voltage": 12,
  "pin_no": "K46",
  "component_name": "点火开关T15",
  "requested_fields": ["pin_definition", "open_voltage", "static_voltage", "idle_voltage"],
  "confidence": 0.92,
  "need_clarify": false
}
```

### 第五步：结构化检索

参数查询服务根据标准化查询对象，执行：

1. 命中文档
2. 命中行
3. 生成证据包

证据包示例：

```json
{
  "matched_doc": {
    "source_knowledge_id": 159,
    "title": "EDC17C53针脚电压(12V系统)",
    "ecu_name": "EDC17C53",
    "system_voltage": 12
  },
  "matched_rows": [
    {
      "component_name": "点火开关T15",
      "ecu_pin_no": "K46",
      "pin_definition": "信号",
      "open_voltage_text": "12V",
      "static_voltage_text": "12V",
      "idle_voltage_text": "12V",
      "remark": ""
    }
  ],
  "ambiguity": null
}
```

### 第六步：输出答案

LLM 最终只基于 evidence package 来组织回答：

- 明确告诉用户命中的 ECU
- 明确告诉用户针脚编号
- 按字段回答
- 如果有备注，附上备注

如果仍不唯一，再调用一次 `ask_user_question` 做一轮澄清。

## 7. 为什么用户口语化输入也能准确映射

核心不是让模型记住所有标题，而是建立“口语 -> 标准查询对象”的归一化链路。

### 7.1 归一化分层

#### A. 显式规则层

负责提取：

- `K46`
- `A-17`
- `12V`
- `24V`
- `开路电压`
- `静态电压`

这一步不依赖 LLM。

#### B. 别名字典层

负责把口语词映射到标准字段：

- `针脚作用` -> `pin_definition`
- `定义` -> `pin_definition`
- `功能` -> `pin_definition`
- `开路电压` -> `open_voltage`
- `静态电压` -> `static_voltage`
- `怠速电压` -> `idle_voltage`

以及 ECU / 零部件的俗称映射。

#### C. 候选召回层

负责把用户说的：

- `EDC17`
- `易控 F17`
- `点火开关`
- `油门踏板`
- `CAN 高`

映射到本地已有的候选文档和候选零部件集合。

#### D. 小范围 LLM 规范化层

LLM 不面对整个数据库，只面对：

- 少量候选 ECU
- 少量候选部件
- 少量候选字段

这样它更容易做准确判断。

## 8. 本地数据库表结构设计

以下新表全部放在当前库 `crs_agent` 中，不放到外部 `decoder_sit`。

原因：

- 当前系统是外部知识库消费者，不应污染源库
- 本地结构化表属于派生投影，不是主业务数据
- 解析策略升级时，需要当前系统自行重建

## 8.1 `param_knowledge_source`

用途：

- 保存外部知识记录的本地镜像元信息
- 保存原始 Markdown 快照
- 判断是否需要重同步、重解析

建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `bigint` | 本地主键 |
| `source_knowledge_id` | `bigint` | 外部 `ai_knowledge.id`，唯一 |
| `source_type_id` | `bigint` | 外部 `type_id` |
| `source_type_code` | `varchar(50)` | 当前首期固定为 `pin_info` |
| `title` | `varchar(500)` | 外部标题 |
| `title_normalized` | `varchar(500)` | 标题归一化 |
| `ecu_name` | `varchar(120)` | 从标题解析出的标准 ECU 名称 |
| `ecu_name_normalized` | `varchar(120)` | ECU 名称归一化 |
| `system_voltage` | `tinyint` | `12/24`，未知可空 |
| `pin_doc_kind` | `varchar(32)` | `pin_voltage` / `pin_definition` / `unknown` |
| `content_format` | `varchar(10)` | `md/text/xml` |
| `content_summary` | `varchar(500)` | 外部摘要 |
| `raw_content` | `mediumtext` | 原始全文快照 |
| `content_hash` | `char(64)` | 内容哈希 |
| `source_status` | `tinyint(1)` | 外部状态 |
| `source_is_deleted` | `tinyint(1)` | 外部删除标志 |
| `source_latest_version` | `int` | 外部最新版本 |
| `source_published_version` | `int` | 外部已发布版本 |
| `source_update_time` | `datetime` | 外部更新时间 |
| `parser_version` | `varchar(32)` | 本地解析器版本 |
| `parse_status` | `varchar(20)` | `ready/failed/pending/skipped` |
| `parse_error` | `varchar(1000)` | 解析错误信息 |
| `parsed_row_count` | `int` | 当前解析行数 |
| `last_synced_at` | `datetime` | 最近同步时间 |
| `created_at` | `datetime` | 创建时间 |
| `updated_at` | `datetime` | 更新时间 |

建议索引：

- `uk_param_source_knowledge_id (source_knowledge_id)`

- `idx_param_source_ecu_name (ecu_name_normalized)`
- `idx_param_source_update_time (source_update_time)`
- `idx_param_source_parse_status (parse_status)`

## 9. 为什么不单独设计一张“AI 可读目录表”

当前阶段不建议新增持久化的 `param_entity_catalog` 表。

原因：

1. 当前 `pin_info` 数量不大
2. ECU / 零部件 / 字段目录都可以由 `param_knowledge_source + param_pin_row + param_alias` 构建
3. 这些目录更适合作为运行时内存索引，而不是持久化主结构

因此当前设计建议：

- 目录不单独落表
- 启动同步完成后重建内存索引

## 10. 运行时内存索引设计

为满足“AI 需要知道当前库里有哪些针脚信息”，不建议每次都扫数据库。

应在服务启动或同步完成后构建内存索引：

### 10.1 文档级索引

- `docs_by_ecu_name`
  - key：`ecu_name_normalized`
  - value：文档列表

- `docs_by_voltage`
  - key：`12` / `24`
  - value：文档列表

- `docs_by_title_token`
  - key：标题 token
  - value：文档列表

### 10.2 行级索引

- `rows_by_doc_pin`
  - key：`(source_knowledge_id, ecu_pin_no_normalized)`
  - value：结构化行列表

- `rows_by_doc_component`
  - key：`(source_knowledge_id, component_name_normalized)`
  - value：结构化行列表

- `rows_by_doc_definition`
  - key：`(source_knowledge_id, pin_definition_normalized)`
  - value：结构化行列表

### 10.3 别名索引

- `field_alias_map`
- `component_alias_map`
- `ecu_alias_map`

这样做的价值是：

- LLM 不必知道整库内容
- 参数查询服务可以先给出小范围候选集
- LLM 只在候选集中做规范化选择

## 11. Query Object 设计

参数查询归一化后的统一对象：

```json
{
  "intent": "PARAM_QUERY",
```

## 12. Evidence Package 设计

参数查询服务输出给 LLM 的统一证据结构：


## 13. 澄清策略设计

首期参数查询建议最多一轮澄清。

触发条件：

1. ECU 候选过多
2. 同一 ECU 同时存在 12V / 24V 版本
3. 用户只给了部件，没给 ECU
4. 用户只说“查一下这个针脚”，但未说明要查定义还是电压

澄清内容建议一次性收齐：

- ECU 候选
- 12V / 24V
- 查什么字段

前端交互建议仍然沿用 `ask_user_question`，但业务归类为 `PARAM_QUERY`。

用户输入：

归一化结果：

- 如果上下文已有当前 ECU，则直接在当前 ECU 内查 `CANH`
- 如果上下文没有 ECU，则必须澄清 ECU

## 15. 首期范围与后续扩展

首期范围：

- 只接 `pin_info`
- 只解决针脚定义、电压、针脚编号类查询

后续可扩展：

- `dataflow`
- `standard_datastream`
- `sensor_pack`

因此本能力命名建议保持泛化：

- 业务：`PARAM_QUERY`
- 领域：`parameter_query`

而不是把能力命名为 `pin_info_query`。

## 16. 最终结论

参数查询能力的关键不是“把外部文档接进来”，而是：

1. 把外部 Markdown 解析成结构化行
2. 用本地投影层承接外部变化
3. 用规则 + 别名 + 候选召回 + 小范围 LLM 规范化，将口语问题映射成标准查询对象
4. 用结构化证据回答用户，而不是用全文猜答案

本设计下，AI 不需要记住数据库所有标题，只需要通过参数查询服务拿到“小而准”的候选集和证据包，就可以稳定完成口语查询映射。
