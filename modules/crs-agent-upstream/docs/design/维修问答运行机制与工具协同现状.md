# 维修问答运行机制与工具协同现状

## 1. 目的

本文说明当前代码中，用户的维修问答请求进入后，系统是如何一步步运行到最终产出回答的。

重点回答 3 个问题：

1. 请求进入后会经过哪些阶段
2. 每个阶段调用哪个 agent、能用哪些工具、在什么条件下切换
3. 当前“工具自由组合、LLM 自主判断缺什么再调用工具或追问用户”的架构，是否已经完全打通


## 2. 总体结论

当前实现已经打通了维修问答主链路：

- 用户问题进入
- 维修类问题识别
- repair gate 答前审查
- 本地维修知识检索
- 信息不足时 ask_user 追问
- 用户补充后恢复上下文
- 再次审查是否可以回答
- 选择最终回答框架
- 输出维修建议

但当前还不是“单一大 agent 持有所有工具，自由组合直到收敛”的完全体 agentloop。

真实状态更接近：

- 一个分阶段的 agentloop
- 不同阶段由不同 agent 承担不同职责
- 每个 agent 能见到的工具集不同
- 运行时仍保留了部分确定性 review / fallback / workflow 分流逻辑

## 4. 入口与第一层分流

### 4.1 统一入口

维修问答请求统一从下面入口进入：

- [backend/app/agent/runtime/service.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/runtime/service.py)
- 方法：`AgentLoopService.process(...)`

这个入口先做几件事：

1. 准备 request 级别的 runtime deps
2. 挂载 case context 和 loop guard
3. 做意图识别
4. 判断是否进入专门工作流


### 4.2 先分流，不是所有问题都直接进维修问答

当前系统会优先判断是否命中两个专门 workflow：

- 文档检索工作流：`_process_doc_search_workflow(...)`
- 参数查询工作流：`_process_parameter_query_workflow(...)`

也就是说，用户说的是维修问题，不代表一定会先进入 repair gate。

如果请求更像：

- “帮我找某份资料”
- “这个针脚定义是什么”
- “某接口参数是多少”

它可能先被分流到 doc search 或 parameter query。

这是当前架构和“完全自由 agentloop”的第一处差异：运行时有明显的前置 workflow 分流。


## 5. 维修问答主链路

### 5.1 什么时候进入 repair gate

入口判断在：

- [backend/app/agent/runtime/service.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/runtime/service.py)
- 方法：`_should_use_repair_answer_gate(...)`

当前进入 repair gate 的典型条件：

- 本轮是维修问答类问题
- 或者本轮是上一轮维修追问 ask_user 的恢复态
- 并且当前可用 `repair_knowledge_service`
- 并且存在 `repair_gate_agent`


### 5.2 repair gate 的职责

repair gate 不是最终回答 agent，它的职责是：

- 判断当前信息是否足够直接回答
- 如果不够，先追问
- 如果够了，再放行进入最终回答阶段

对应 agent：

- 文件：[backend/app/agent/runtime/factory.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/runtime/factory.py)
- 方法：`create_repair_gate_agent(...)`
- agent 名称：`crs_repair_pre_answer_gate`


### 5.3 repair gate 当前可用工具

当前 repair gate 可见工具是受限的，不是全量工具：

- `ask_user_question`
- `lookup_repair_knowledge_titles`
- `get_repair_knowledge_context`

这意味着 gate 当前可以：

- 追问用户
- 在本地维修知识库里查标题
- 载入本地维修知识上下文

但它当前不能直接：

- 调 `query_parameters`
- 调 `dtc_diagnosis`
- 调 `lookup_ecu_candidates`

这也是当前“工具自由组合尚未完全打通”的关键事实之一。


## 6. repair gate 之后不是直接信 LLM，还会再 review

repair gate 输出后，运行时还会再做一层 review。

位置：

- [backend/app/agent/domain/repair_knowledge/review.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/domain/repair_knowledge/review.py)

核心方法：

- `review_repair_answer_gate(...)`
- `review_repair_answer_gate_async(...)`

它的作用不是回答，而是做运行时的二次判定：

- 是否还缺关键字段
- 缺的是哪些字段
- 当前是否真的可以放行

也就是说，当前实现不是“gate LLM 说能答就能答”，而是：

- gate LLM 先判断
- runtime review 再复核

这也是当前架构不是完全自由型 agentloop 的另一处体现。


## 7. ask_user 是怎么生成的

### 7.1 ask_user 不是直接把原始问题扔给前端

无论是：

- gate 主动 ask_user
- 还是 review 判定后生成 ask_user

最后都会进入统一规范化过程，把它变成前端能渲染的结构化表单。

关键位置：

- [backend/app/agent/runtime/service.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/runtime/service.py)
  - `_normalize_runtime_ask_user_question_async(...)`
- [backend/app/agent/adapters/repair_knowledge_followup_adapter.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/adapters/repair_knowledge_followup_adapter.py)
  - `normalize_ask_user_question_async(...)`
  - `build_ask_user_question_async(...)`


### 7.2 ask_user 的组成

最终前端看到的 ask_user，通常会被整理成：

- question
- ask_reason
- field_groups
- form
- 每个字段的 options / placeholder / hint / required_level

也就是说，前端展示的并不是 LLM 原始输出，而是“LLM 输出 + adapter 规范化 + 表单化”的组合结果。


## 8. 恢复态是怎么接上的

这是维修问答多轮追问的关键部分。

### 8.1 用户答完 ask_user 后

当用户提交 `ask_user_answer`，运行时会：

1. 从 deferred state 恢复当时的上下文
2. 把本轮用户补充写入 case context
3. 重新构造 prompt

关键位置：

- [backend/app/agent/runtime/service.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/runtime/service.py)
  - `_prepare_run_state(...)`
  - `_record_case_context_user_answer(...)`


### 8.2 恢复态 prompt 会显式告诉 agent 什么已经答过

运行时会自动给恢复态补一个专门的 prompt 包装：

- `[REPAIR_FOLLOWUP_RESUME]`

里面会带上：

- 原始问题
- 已回答字段 key
- 用户补充摘要
- 已回答内容
- 明确禁止重复问已回答字段

关键位置：

- [backend/app/agent/runtime/service.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/runtime/service.py)
  - `_build_user_prompt_with_case_context(...)`
  - `_decorate_repair_followup_resume_prompt_if_needed(...)`


### 8.3 当前恢复态机制已经基本打通

现在恢复态已经能做到：

- 保留原始问题
- 合并历史回答
- 在下一轮 gate/review 中继续使用这些信息
- 显式约束不要重复问

但它仍然不是百分之百由 LLM 自主记忆完成，而是 runtime 主动把这些信息写进 prompt。


## 9. 什么情况下进入最终回答阶段

如果 repair gate + review 共同判定：

- 当前信息足够回答

就会进入 render 阶段，而不是直接把 gate 的文本发给用户。


## 10. render 阶段是两步，不是一步

### 10.1 第一步：先选回答框架

对应 agent：

- 文件：[backend/app/agent/runtime/factory.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/runtime/factory.py)
- 方法：`create_repair_render_planner_agent(...)`
- 名称：`crs_repair_render_planner`

它的职责：

- 决定当前回答该用什么 frame
- 决定 direct / standard / playbook 哪种深度

常见 frame 包括：

- `dtc_diagnosis`
- `symptom_diagnosis`
- `spec_answer`
- `principle_explanation`
- `location_identification`
- `operation_guide`

它不调用工具，只做结构化规划。


### 10.2 第二步：真正生成最终回答

对应 agent：

- 文件：[backend/app/agent/runtime/factory.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/runtime/factory.py)
- 方法：`create_repair_renderer_agent(...)`
- 名称：`crs_repair_answer_renderer`

当前 renderer 能用的工具：

- `lookup_ecu_candidates`
- `dtc_diagnosis`
- `lookup_repair_knowledge_titles`
- `get_repair_knowledge_context`
- `query_parameters`

但它被明确禁止：

- `ask_user_question`

也就是说，renderer 只能回答，不能继续追问。

## 12. 当前哪些部分已经打通

当前已经打通的部分：

1. 维修类问题进入 repair gate 的主链路
2. 本地维修知识工具与 ask_user 的组合调用
3. ask_user 恢复态上下文继承
4. repair gate -> review -> renderer 的完整闭环
5. renderer 侧结合知识/报码/参数工具输出最终回答
6. 参数查询 workflow 自己的 clarify -> ask_user -> resume 闭环


## 13. 当前哪些部分还没有完全打通

### 13.1 还不是单一自由组合的大 agent

当前是：

- 多阶段 agent
- 各阶段工具权限不同
- runtime 有显式 workflow 分流

不是：

- 单个 LLM 拿全工具后自由规划、自由切换、自由组合直到收敛


### 13.2 repair gate 工具仍然受限

repair gate 现在不能直接调用：

- `query_parameters`
- `dtc_diagnosis`
- `lookup_ecu_candidates`

所以它还不能真正做到“根据缺什么信息自由组合全部系统工具”。


### 13.3 renderer 虽然工具更多，但不能再 ask_user

这意味着当前系统把“追问”和“最终回答”强行拆成了两个阶段。

优点是稳定；
缺点是它不完全符合“LLM 在必要时随时继续追问”的自由 agentloop 理念。


### 13.4 仍保留确定性 runtime review / fallback

当前代码里，下面两块仍然承担了较强的运行时控制职责：

- [backend/app/agent/domain/repair_knowledge/review.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/domain/repair_knowledge/review.py)
- [backend/app/agent/adapters/repair_knowledge_followup_adapter.py](/Users/zhangjiexiang/VisualStudioCodeProject/crs-agent/backend/app/agent/adapters/repair_knowledge_followup_adapter.py)

这意味着：

- LLM 不是唯一决策者
- 运行时仍然会在一些地方强行改写追问结构、候选项和是否放行


### 13.5 registry 中的工具不等于所有 agent 都能用

例如工具注册表里有：

- `search_circuit_diagram`

但当前维修主链路相关 agent 并没有实际把它暴露出来使用。

所以“工具注册了”不等于“已经打通到维修问答 agent 可自由调用”。


## 14. 最终结论

当前维修问答运行机制，已经实现了：

- 请求识别
- 维修知识检索
- 追问
- 多轮恢复
- 再审查
- 最终渲染回答

这一整条主链路。

但从架构角度看，当前仍然是：

- 分流式 workflow + 分阶段多 agent + 受限工具集

而不是：

- 统一大 agent + 全工具自由组合 + 完整自主规划

因此，当前状态应定义为：

**维修问答主链路已打通，但“工具自由组合、LLM 完全自主调度”的目标尚未完全实现。**
