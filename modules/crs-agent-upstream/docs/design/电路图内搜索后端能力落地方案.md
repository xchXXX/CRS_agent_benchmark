# 电路图内搜索后端能力落地方案

## 1. 目标

本阶段只做后端能力，不改用户端前端。

目标是在现有文档搜索结果上增加图内搜索增强：

```text
现有 doc_search results
  -> 用 filename 匹配解析库 items.name
  -> 拿 items.latest_pdf_id
  -> 调外部图内搜索接口
  -> 压缩多命中结果
  -> 把 body_search 挂回原 result
```

现有文档召回、澄清、排序逻辑不改。

## 2. 后端包结构

新增：

```text
backend/app/agent/domain/circuit_body_search/
  __init__.py
  models.py
  config.py
  keyword.py
  parsed_doc_resolver.py
  search_client.py
  reducer.py
  enhancer.py
```

职责：

```text
models.py
  定义内部数据结构，不绑定前端展示。

config.py
  从 config_service/settings 读取图内搜索配置。

keyword.py
  解析 body_keyword。当前优先使用 query planner 输出的 body_keyword，再回退到历史 query 字段。

parsed_doc_resolver.py
  批量 filename -> parsed document -> latest_pdf_id / latest_result_path / url_raw_sample。

search_client.py
  调外部 POST /api/search。

reducer.py
  把外部接口的大量命中压缩成一个后端结果摘要。

enhancer.py
  编排 resolver/client/reducer，把 body_search 写回 doc_search result。

preview_token.py
  生成和校验局部图预览签名 token，避免把解析结果路径作为未签名参数暴露给前端。

preview_renderer.py
  根据签名 payload 读取解析页图或原始 PDF，按命中坐标裁出局部图。
```

## 3. 数据模型

建议模型：

```python
class ParsedCircuitDocument(BaseModel):
    item_id: str
    name: str
    latest_pdf_id: str
    latest_result_path: str
    url_raw_sample: str = ""
    updated_at: datetime | None = None


class CircuitBodyBestHit(BaseModel):
    hit_id: str
    page_index: int
    page_number: int
    snippet: str = ""
    highlight_boxes_px: list[list[float]] = Field(default_factory=list)
    preview_image_url: str = ""


class CircuitBodySearchSummary(BaseModel):
    status: Literal["hit", "no_hit", "unsupported", "failed"]
    reason: str = ""
    match_source: str = "filename"
    pdf_id: str = ""
    keyword: str = ""
    source_pdf_url: str = ""
    viewer_url_type: str = ""
    raw_hit_count: int = 0
    page_hit_count: int = 0
    best_hit: CircuitBodyBestHit | None = None
    more_hits_count: int = 0
```

`status` 含义：

```text
hit          找到解析文档，并且图内搜索有命中
no_hit       找到解析文档，但图内搜索没有命中
unsupported 该文档没有解析结果，不能做图内搜索
failed       解析库或外部接口失败
```

本阶段前端不展示，但后端 response 可以先透出该字段，方便接口验证。

## 4. 解析库匹配

### 4.1 文件名来源

从现有搜索结果取：

```text
result.filename || result.title
```

归一化：

```python
filename = filename.strip()
filename = re.sub(r"\.(pdf|PDF)$", "", filename)
```

### 4.2 批量查询

不要对 TopN 每条结果单独查库。第一版用批量 SQL，一次拿回每个文件名的最新解析记录：

```sql
WITH ranked AS (
  SELECT
    item_id,
    name,
    latest_pdf_id,
    latest_result_path,
    latest_status,
    updated_at,
    ROW_NUMBER() OVER (
      PARTITION BY name
      ORDER BY updated_at DESC NULLS LAST
    ) AS rn
  FROM items
  WHERE name = ANY(:filenames)
    AND latest_status = 'completed'
    AND latest_pdf_id <> ''
    AND latest_result_path <> ''
)
SELECT
  item_id,
  name,
  latest_pdf_id,
  latest_result_path,
  updated_at
FROM ranked
WHERE rn = 1;
```

返回：

```text
dict[normalized_filename, ParsedCircuitDocument]
```

### 4.3 PostgreSQL 连接

当前 `backend/pyproject.toml` 只有 `sqlalchemy`，没有 PostgreSQL driver。

建议新增依赖：

```text
psycopg[binary]>=3.2.0
```

连接方式：

```text
sqlalchemy.create_engine("postgresql+psycopg://...")
```

连接策略：

```text
connect_timeout 使用配置
pool_pre_ping=True
pool_size=2
max_overflow=2
只读查询，不执行任何写操作
```

## 6. 命中压缩

外部接口返回的是召回集，不直接给前端。

第一版 reducer 规则：

```text
1. 读取 data.results
2. 按 page_index 分组
3. 选择命中数量最多的页
4. 同页内选择 reading_order 最小的命中
5. bounding_box 转成 highlight_boxes_px
6. raw_hit_count = 外部 total_matches 或 results 数
7. page_hit_count = 有命中的页数
8. more_hits_count = page_hit_count - 1
```

输出只保留一个 `best_hit`，避免接口结果过大。

后续如果需要更智能选择，再在 reducer 后增加 LLM reranker，不影响 resolver/client/enhancer。

### 7.2 service 接入点

修改：

```text
backend/app/agent/runtime/service.py
```

接入位置：

```text
AgentLoopService._execute_doc_search_workflow(...)
  search_envelope status ok
  analyze_ambiguity 不需要澄清
  validity.has_valid_results 不是 false
  _build_documents_response_from_envelope 之前
```

建议加私有方法：

```python
async def _enhance_doc_search_with_circuit_body_search(
    self,
    *,
    active_deps: AgentRuntimeDeps,
    search_data: dict[str, Any],
    workflow_state: DocSearchWorkflowRunState,
) -> None:
    enhancer = getattr(active_deps, "circuit_body_search_enhancer", None)
    if enhancer is None:
        return

    keyword = resolve_circuit_body_keyword(
        search_data=search_data,
        fallback_query=workflow_state.query,
    )
    if not keyword:
        return

    search_data["results"] = await enhancer.enhance(
        results=search_data.get("results") or [],
        body_keyword=keyword,
        max_docs=3,
    )
```

`body_keyword` 的来源由资料搜索 planner 前置解析：

```json
{
  "input_mode": "text|text_image|image",
  "primary_query": "用于资料库标题/文件名搜索的词",
  "queries": [
    {"query": "候选资料搜索词", "intent": "doc_search", "confidence": 0.9}
  ],
  "body_keyword": "用于文档内部定位的部件/针脚/线路词",
  "body_keyword_confidence": 0.0,
  "rationale": "规划原因"
}
```

`primary_query` 和 `queries` 用于资料召回；`body_keyword` 会写入最终 `search_data` 和 clarify 快照，图内搜索增强优先用该字段。

然后在原流程中插入：

```python
await self._enhance_doc_search_with_circuit_body_search(
    active_deps=active_deps,
    search_data=search_data,
    workflow_state=workflow_state,
)
```

## 8. Enhancer 编排

`CircuitBodySearchEnhancer.enhance(...)`：

```text
1. 读取 enabled；关闭则原样返回 results
2. 取 TopN，默认 3
3. 提取并归一化 filename
4. resolver.resolve_many(filenames)
5. 对匹配到 latest_pdf_id 的文档并发调用 search_client
6. reducer.reduce(raw_response)
7. 命中时把 `url_raw_sample` 作为 `source_pdf_url`，并用 `latest_result_path/source_pdf_url/page_index/highlight_boxes_px` 生成 `preview_image_url`
8. 给对应 result 增加 body_search
9. 未匹配解析库的 result 不挂字段，或挂 unsupported
10. 任一子步骤失败只影响该 result，不抛出到 doc_search 主流程
```

建议第一版：

```text
没有 hit 的结果不挂 body_search
只有 hit 才挂 body_search
```

这样不会污染现有前端，也便于后续只展示有效入口。

## 8.1 局部图预览接口

搜索响应中的 `body_search.best_hit.preview_image_url` 指向后端图片接口：

```text
GET /chat/api/circuit-body-search/preview/{token}
```

返回：

```text
Content-Type: image/png
Cache-Control: private, max-age=86400
```

token 为后端签名载荷，内部包含：

```json
{
  "pdf_id": "items.latest_pdf_id",
  "latest_result_path": "items.latest_result_path",
  "source_pdf_url": "items.url_raw_sample",
  "page_index": 0,
  "highlight_boxes_px": [[100, 200, 300, 260]]
}
```

渲染顺序：

1. 优先读取 `latest_result_path` 对应解析 JSON，查找 `page_metadata.image_path/image_filename` 并裁图。
2. 如果解析页图不可读，则用 `source_pdf_url` 下载 PDF，按解析坐标默认 600 DPI 渲染对应页的裁剪区域。
3. 裁剪区域基于 `highlight_boxes_px` 加边距并限制输出尺寸。
4. token 非法、过期、资源不可读时返回 404，不影响文档搜索主流程。

## 9. 缓存

第一版不建新表。

可以复用现有 `DocSearchCacheStore` 做短缓存：

```text
filename resolver cache:
  circuit_body:resolver:v1:{normalized_filename}

body search cache:
  circuit_body:search:v1:{pdf_id}:{sha256(keyword)}
```

缓存只用于减少重复请求，不能作为唯一数据源。

建议先实现：

```text
resolver 结果缓存：可以做
外部搜索结果缓存：可选
```
