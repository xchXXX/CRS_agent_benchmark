# 电路图内搜索交互设计和流程

## 1. 背景

当前资料搜索主要基于文档名称、标签、层级路径等文档级信息，搜索结果最多定位到文档级别。实际使用中，用户通常不是只想知道“哪份电路图可能相关”，而是希望直接看到电路图中的目标位置，例如某个传感器、接头、针脚、信号线或电路标题。

部分 PDF 文档已经被解析为带坐标信息的结构化结果，外部服务也具备“搜索指定文件内部元素并返回坐标”的能力。因此，本功能不再重复实现图内全文检索，而是基于现有文档搜索结果，对候选文档调用外部图内搜索服务，并将命中坐标转换为可展示、可定位的用户体验。

## 2. 目标

### 2.1 用户目标

- 用户搜索资料后，仍然看到熟悉的文档搜索结果列表。
- 如果某个电路图内部存在精确命中，结果卡片上给出明确提示。
- 用户点击提示后，可以展开查看局部大图。
- 用户可以选择查看完整文档，或直接定位到搜索命中的位置。

### 2.2 系统目标

- 保持现有文档搜索链路稳定，不打断已有搜索体验。
- 图内搜索作为文档搜索后的增强能力，只对候选文档做精查。
- 避免对全库图内搜索，控制外部服务调用成本和响应时间。
- 局部图按需加载，不在搜索首屏加载所有图片。
- 后续可以平滑扩展到更多文档类型和更复杂的预览定位能力。

## 3. 设计原则

1. 先粗查，再精查  
   现有文档级搜索负责找到候选电路图，外部图内搜索只在候选文档内做精确定位。

2. 列表保持轻量  
   搜索结果列表默认不展示多张局部图，只展示“图内精确命中”的提示。

3. 图片懒加载  
   用户点击展开后，再请求局部图和更完整的命中信息。

4. 点击行为明确  
   展开区域提供两个动作：“查看完整电路图”和“定位到命中位置”。

5. 兼容未解析文档  
   未解析或不支持图内搜索的文档继续按原来的文档卡展示。

## 4. 推荐交互

### 4.1 默认搜索结果

搜索结果默认保持现有样式。对于有图内搜索命中的结果，在文档卡片中增加一个轻量提示行：

```text
福康_F3.8_发动机维修手册-2
标签：康明斯 / F3.8 / 电路图

[图内精确命中] 第403页 · 2处命中 · 点击展开
```

推荐文案：

```text
图内精确命中 · 点击展开
```

或：

```text
命中图内位置 · 查看局部图
```

不建议使用过长文案，例如“此电路图有图内精确信息，点击查看”，因为结果列表空间有限，移动端也容易换行。

### 4.2 展开后的局部区域

用户点击“图内精确命中”后，在当前结果卡片内下拉展开，或在移动端使用底部抽屉展示。

```text
命中第403页
转速表电路 ... ECM ... OEM线束中的转速表信号导线

[局部大图]

[查看完整电路图]   [定位到命中位置]
```

按钮含义：

- 查看完整电路图：打开文档预览，跳到命中页，但保留完整页面视角。
- 定位到命中位置：打开文档预览，跳到命中页并滚动/缩放到命中坐标区域，同时高亮命中框。

如果当前预览器暂时无法做坐标级定位，第一版可以先做到：

- 查看完整电路图：打开文档。
- 定位到命中位置：打开文档并跳到命中页。

后续再补充坐标滚动和高亮。

### 4.3 多个候选结果

不建议前 5 个候选都默认展示局部图。推荐规则：

- 每个结果卡默认只展示命中提示，不展示图片。
- 同一时间最多展开一个结果的局部图。
- 每个文档默认展示最佳 1 个命中。
- 如果有多个命中，提示“还有 N 处命中”，展开后可切换查看。

示例：

```text
[图内精确命中] 第403页 · 还有2处命中 · 点击展开
```

## 5. 搜索流程

### 5.1 主流程

```text
用户输入
  ↓
意图识别：资料搜索
  ↓
现有文档级搜索
  ↓
返回 Top N 文档候选
  ↓
筛选支持图内搜索的候选文档
  ↓
调用外部图内搜索服务
  ↓
获得页码、命中文本、元素坐标、命中分数
  ↓
将图内命中挂到对应文档结果上
  ↓
前端展示文档列表和“图内精确命中”提示
  ↓
用户点击展开
  ↓
加载局部大图
  ↓
用户点击查看完整文档或定位到命中位置
```

### 5.2 资源最优策略

图内搜索应该放在文档搜索之后，而不是搜索前或全库搜索。

推荐第一版策略：

```text
文档搜索 Top5
  ↓
只对 Top3 中支持图内搜索的文档调用外部图内搜索
  ↓
每个文档最多取 1-2 个最佳命中
  ↓
搜索结果只返回命中摘要和坐标，不默认生成局部图
  ↓
用户点击展开时再生成或加载局部图
```

如果 Top3 都没有图内命中，可以按需扩展：

- 查 Top4-Top5。
- 或在前端提供“查看更多图内命中”入口。
- 或只保持普通文档搜索结果，不继续追加外部调用。

### 5.3 同步与异步选择

#### 同步增强

后端等待 Top3 图内搜索完成后，再一次性返回搜索结果。

优点：

- 前端逻辑简单。
- 用户看到结果时已经带有图内命中提示。

缺点：

- 首屏响应时间会被外部图内搜索拖慢。

适合：

- 外部图内搜索延迟稳定在较低范围内。
- Top3 调用整体耗时可控。

#### 异步增强

先返回普通文档搜索结果，再异步补充图内命中状态。

```text
先返回文档搜索结果
  ↓
前端立即展示列表
  ↓
后台继续查 Top3 图内命中
  ↓
命中完成后更新对应结果卡片
```

优点：

- 首屏最快。
- 外部服务慢时不阻塞基础搜索体验。

缺点：

- 需要前端支持结果卡片状态更新。
- 需要后端提供轮询、SSE 事件或补充查询接口。

适合：

- 外部图内搜索耗时不稳定。
- 需要保证搜索首屏响应速度。

第一版可以先做同步增强，若耗时明显，再改为异步补充。

## 6. 外部图内搜索接入位置

外部图内搜索服务只负责：

- 在指定文件内部搜索用户 query。
- 返回命中页码。
- 返回命中文本或 snippet。
- 返回命中元素坐标。
- 返回命中分数。

本系统负责：

- 决定调用哪些候选文档。
- 将外部命中挂到文档搜索结果上。
- 根据坐标生成局部图展示区域。
- 生成文档预览定位参数。
- 控制前端展开、懒加载和按钮行为。

## 7. 命中过多时的收敛与 LLM 重排

### 7.1 问题

外部图内搜索接口可能返回大量命中。例如用户搜索“涡轮增压器”时，单本文档可能返回 100 多个元素命中，分布在几十页。直接展示这些命中会导致：

- 用户无法快速判断哪个位置最相关。
- 前端结果卡片过重。
- 局部图生成成本上升。
- 系统需要维护大量业务词表来判断哪些命中更像“电路相关”，复杂度会持续增加。

因此，外部搜索结果不能直接作为展示结果。它应该被视为召回集，本系统需要先把召回集压缩为少量候选，再决定默认展示哪一个。

### 7.2 总体策略

推荐采用：

```text
机械聚合 + 候选证据包 + LLM 重排 + 可回退交互
```

职责划分：

- 程序层负责稳定、低成本的机械处理：分组、聚类、截取上下文、控制候选数量。
- LLM 负责理解用户 query 与候选证据之间的关系，选择最适合作为默认展示的命中。
- 前端保留“其他命中”入口，避免系统默认推荐不符合用户真实意图时无路可退。

不要采用：

```text
纯规则大词表
```

因为实际维修资料中的表达会持续变化，长期维护大词表会显著提升局部复杂度。

也不要采用：

```text
LLM 直接读取全部外部命中
```

因为结果数量可能很多，且大部分元素命中是重复或弱相关的。LLM 应该只读取经过机械压缩后的候选证据包。

### 7.3 执行链路

```text
外部图内搜索返回 results
  ↓
按 page_index 分组
  ↓
同页按坐标聚成区域候选
  ↓
从解析 JSON 中取命中附近 OCR 文本
  ↓
生成候选证据包
  ↓
限制候选数量，例如 15-20 个
  ↓
调用 LLM 做 rerank
  ↓
返回默认推荐 Top1、备选方向、是否需要二次收敛
```

注意：这里不重新扫描 700 页全文，只处理外部接口已经返回的命中页和命中元素。对于 100-300 条命中，机械聚合通常只是内存数组处理，成本很低。

### 7.4 区域候选如何生成

外部接口返回的是元素级命中，例如：

```json
{
  "match_id": "p512_e5_c0",
  "page_index": 512,
  "element_index": 5,
  "reading_order": 6,
  "matched_text": "涡轮增压器",
  "context": "涡轮增压器执行器",
  "bounding_box": {
    "x_min": 571,
    "y_min": 500,
    "x_max": 1151.625,
    "y_max": 670
  }
}
```

同一页可能存在多个命中。程序需要先把这些命中按空间位置合并成区域候选：

```text
同页命中
  ↓
按 bbox 中心点排序
  ↓
距离较近的合并为同一区域
  ↓
距离较远的保留为多个候选
```

第一版可以使用简单策略：

- 按 `page_index` 分组。
- 同页内按 `y_center` 排序。
- `y_center` 距离小于页面高度一定比例的命中合并，例如 10%-12%。
- 横向距离很近的命中也可以合并。
- 每个区域生成一个 `region_bbox_px`。

这样一页内可能得到：

```text
区域 1：页上方标题或说明
区域 2：中间图纸标注
区域 3：底部步骤说明
```

LLM 接触的是区域候选，而不是一页里混杂在一起的所有命中。

### 7.5 候选证据包如何生成

候选证据包不是主观摘要，而是结构化原始证据。它应该尽量保留 LLM 做选择所需的信息：

- 命中文本。
- 外部接口返回的 `context`。
- 命中附近的 OCR 原文。
- 页标题或区域标题。
- 命中数量。
- 页码。
- 页内位置。
- 坐标范围。

示例：

```json
{
  "candidate_id": "p512_r1",
  "page_index": 512,
  "page_number": 513,
  "match_count": 3,
  "exact_matches": ["涡轮增压器", "涡轮增压器执行器"],
  "hit_contexts": [
    "涡轮增压器执行器",
    "新的涡轮增压器执行器空..."
  ],
  "page_heading": "涡轮增压器执行器",
  "nearby_text": "涡轮增压器执行器 ... 空气管 ... 检查 ... 更换 ...",
  "position": "页面上部",
  "bbox_norm": [0.11, 0.08, 0.31, 0.24]
}
```

候选证据包的关键原则：

- 不要只写“这里提到了涡轮增压器”。
- 不要提前替 LLM 判断“这是电路相关”或“这是维修相关”。
- 尽量提供原始文本证据，让 LLM 根据用户 query 自己判断。

### 7.6 附近文本如何截取

附近文本建议由三部分合成，互相兜底：

1. 外部接口返回的 `context`。  
   这是最直接的命中上下文，但通常较短。

2. 解析 JSON 中按 `reading_order` 取命中元素前后若干 OCR 元素。  
   例如前后各 8-15 个元素。适合普通说明文本。

3. 解析 JSON 中落在区域 bbox 外扩范围内的 OCR 元素。  
   例如区域框外扩 10%-15% 后，取其中所有 OCR 文本。适合电路图这类二维布局，因为阅读顺序不一定可靠。

合成后需要去重、截断，避免给 LLM 太长输入。建议每个候选的 `nearby_text` 控制在 200-400 个中文字符以内。

### 7.7 候选数量控制

如果外部接口返回 126 个元素命中，可能会变成：

```text
126 个元素命中
  -> 59 个页命中
  -> 60-80 个区域候选
  -> 15-20 个候选证据包给 LLM
```

候选数量控制不应该依赖复杂语义词表，只做轻量筛选：

- 每页最多保留 1-2 个区域候选。
- 优先保留命中数量更多的区域。
- 优先保留 `context` 更完整的区域。
- 去掉高度重复的候选。
- 如果候选仍然过多，只取前 15-20 个给 LLM。

这一步的目标是控成本，不是做最终语义判断。

### 7.8 LLM 重排任务

LLM 不负责回答用户问题，只负责从候选证据包中选择最适合作为默认局部图展示的候选。

Prompt 要点：

```text
你是电路图搜索结果排序器。
用户输入：{query}

下面是同一文档内的图内搜索候选。每个候选包含页码、命中次数、命中文本上下文和位置。
请选择最适合作为默认局部图展示的候选。

要求：
1. 不要编造候选中没有的信息。
2. 如果用户输入很宽泛，不要假装唯一正确；选择最代表性的候选，并给出 2-3 个备选方向。
3. 如果多个候选意图差异明显，返回 needs_refinement=true。
4. 只输出 JSON，不要输出自然语言。
```

推荐输出结构：

```json
{
  "top_candidate_id": "p512_r1",
  "confidence": 0.82,
  "reason": "候选区域包含更具体的涡轮增压器执行器信息，适合作为默认展示。",
  "alternative_groups": [
    {
      "label": "结构说明",
      "candidate_ids": ["p486_r1"]
    },
    {
      "label": "检查/更换",
      "candidate_ids": ["p501_r2", "p502_r1"]
    }
  ],
  "needs_refinement": true,
  "refinement_options": ["执行器", "结构说明", "检查更换", "全部命中"]
}
```

### 7.9 前端展示方式

如果 LLM 给出推荐命中，前端仍然保持轻量提示：

```text
图内精确命中 · 推荐第513页 · 共59页相关 · 点击展开
```

展开后显示：

```text
推荐命中：第513页
推荐原因：附近包含“涡轮增压器执行器”等更具体信息

[局部大图]

[查看完整电路图] [定位到命中位置]

其他方向：结构说明 / 检查更换 / 全部命中
```

注意文案不要暗示系统已经找到唯一正确答案。推荐使用：

```text
推荐命中
```

不要使用：

```text
唯一命中
```

这样即使默认推荐不符合用户真实意图，用户也能通过“其他方向”或“全部命中”快速回退。

### 7.10 什么时候调用 LLM

LLM 重排不是每次都必须调用。推荐触发条件：

- 元素命中数超过 5。
- 或页级命中数超过 3。
- 或同一页存在多个相距较远的候选区域。

可以直接跳过 LLM 的情况：

- 只有 1 个命中。
- 只有 1 页命中，且区域候选只有 1 个。
- 外部接口返回的命中数量很少，直接展示即可。

超时和降级：

- LLM rerank 设置短超时。
- LLM 失败时使用机械排序的 Top1。
- 不因为 LLM 失败阻塞普通文档搜索结果。

### 7.11 准确性边界

系统不能保证默认展示的就是用户最想要的内容，尤其当用户只输入宽泛部件名时，例如“涡轮增压器”。合理目标是：

```text
默认展示最可能的 1 个
同时保留其他命中入口
用户发现不对时能快速切换或缩小范围
```

因此，最终体验必须包含：

- 推荐命中。
- 推荐依据。
- 其他方向。
- 查看全部命中。
- 必要时引导用户补充更具体关键词。

## 8. 数据结构建议

### 8.1 搜索结果中的轻量命中摘要

搜索首屏不需要返回局部图，只需要返回轻量字段：

```json
{
  "file_id": "xxx",
  "filename": "福康_F3.8_发动机维修手册-2",
  "score": 0.86,
  "pic_folder_url": "https://...",
  "body_search": {
    "status": "hit",
    "source_pdf_url": "http://static2.51gonggui.com/wps/file/xxx.pdf",
    "viewer_url_type": "raw_pdf",
    "hit_count": 2,
    "best_hit": {
      "hit_id": "hit_xxx_402_1",
      "page_index": 402,
      "page_number": 403,
      "snippet": "转速表电路 ... ECM ... OEM线束中的转速表信号导线",
      "highlight_boxes_px": [[288, 166, 699, 278]],
      "preview_image_url": "/chat/api/circuit-body-search/preview/xxx",
      "score": 0.91
    }
  }
}
```

`body_search.status` 建议取值：

- `unsupported`：文档不支持图内搜索。
- `pending`：图内搜索仍在处理中。
- `hit`：存在图内命中。
- `miss`：已查过，但没有图内命中。
- `failed`：图内搜索失败。

### 8.2 展开局部图详情

用户点击展开时，优先直接加载搜索结果里的局部图 URL：

```text
GET /chat/api/circuit-body-search/preview/{token}
```

该接口返回 `image/png`。token 由后端签名，内部包含 `latest_result_path`、`source_pdf_url`、`page_index` 和 `highlight_boxes_px`，前端不需要也不应该拼接这些原始路径。

如果后续需要更完整的命中详情，可以再增加详情接口：

```text
GET /chat/api/document-body-hits/{hit_id}
```

返回：

```json
{
  "hit_id": "hit_xxx_402_1",
  "file_id": "xxx",
  "page_index": 402,
  "page_number": 403,
  "snippet": "转速表电路 ... ECM ... OEM线束中的转速表信号导线",
  "thumbnail_url": "/chat/api/document-body-hits/hit_xxx_402_1/crop",
  "target": {
    "coordinate_space": "page_image_px",
    "page_width_px": 5100,
    "page_height_px": 6600,
    "crop_bbox_px": [240, 140, 1800, 1800],
    "highlight_boxes_px": [
      [288, 166, 699, 278]
    ]
  }
}
```

坐标协议必须明确：

- 坐标基于 PDF 原始点坐标，还是页面图片像素坐标。
- 是否包含 OCR padding。
- 页码是 0 开始还是 1 开始。
- 页面旋转、缩放后如何换算。

推荐内部统一使用：

```text
page_index: 0 开始
page_number: 1 开始，仅用于展示
coordinate_space: page_image_px
bbox: [x1, y1, x2, y2]
```

## 9. 局部图生成策略

外部服务返回元素坐标后，本系统按以下规则生成局部图：

1. 合并同页相近命中框。
2. 根据命中框向四周扩展，避免只截到几个字。
3. 如果命中在页眉或标题区域，向下扩展更多区域，带出图纸主体。
4. 限制最大裁剪区域，避免局部图退化成整页图。
5. 返回局部图 URL，前端展开时加载。

第一版可以动态生成裁剪图。后续如果访问量增加，可以对 `hit_id + crop_bbox` 做缓存。

## 12. 性能控制

建议默认限制：

- 文档搜索候选：Top5。
- 图内搜索候选：Top3。
- 每个文档返回命中：最多 1-2 个。
- 默认展开图片：0 个。
- 用户点击后懒加载局部图。
- 局部图缓存：按需后置。

超时策略：

- 单个文档图内搜索设置短超时。
- 外部服务失败时不阻塞普通文档搜索。
- 若同步增强总耗时超过阈值，直接返回普通搜索结果。

## 13. 分阶段落地

### Phase 1：结果增强与展开

- 文档搜索后对 Top3 调用外部图内搜索。
- 搜索结果增加 `body_search` 轻量字段。
- 前端结果卡增加“图内精确命中”提示。
- 点击展开后展示局部图。
- 提供“查看完整电路图”和“定位到命中位置”按钮。

### Phase 2：懒加载与缓存

- 搜索首屏只返回 `hit_id` 和摘要。
- 展开时请求命中详情和局部图。
- 对局部图做缓存。
- 同一时间只展开一个命中卡片。

### Phase 3：预览定位增强

- 建设自控文档预览页。
- 支持按页跳转。
- 支持坐标级滚动和高亮。
- 支持命中区域缩放。

### Phase 4：异步补充

- 普通搜索结果先返回。
- 图内搜索异步补充。
- 通过 SSE、轮询或补充接口更新结果卡片状态。

### Phase 5：命中过多收敛与 LLM 重排

- 对外部图内搜索大量命中做页级和区域级聚合。
- 为每个区域生成候选证据包。
- 只将少量候选证据包交给 LLM 做 rerank。
- 前端展示推荐命中、推荐依据和其他方向。
- LLM 失败时降级为机械排序 Top1。

## 14. 推荐结论

图内搜索不应该放在搜索链路最前面，也不应该全库执行。最合适的位置是在现有文档级搜索之后，对 Top 候选文档做精查。

最终用户体验保持为：

```text
文档搜索结果列表保持不变
  ↓
有图内命中的文档显示“图内精确命中”
  ↓
点击后展开局部大图
  ↓
用户选择查看完整电路图或直接定位命中位置
```

这种设计能在不破坏当前搜索体验的前提下，把“找到文档”升级为“找到文档中的目标位置”，同时控制外部搜索调用、图片加载和预览定位的资源成本。
