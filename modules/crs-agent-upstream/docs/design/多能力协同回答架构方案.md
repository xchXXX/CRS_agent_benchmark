# 基于当前项目的多能力协同回答架构方案

## 1. 文档目标

这份文档回答一个具体问题：

当前项目已经具备资料搜索、故障诊断、维修知识、`ask_user` 等能力，下一步如果要演进成“多个能力配合回答同一个用户问题”的系统，应该怎么改。

这里讨论的是架构演进方案，不讨论具体代码实现细节。

## 2. 要解决的核心问题

当前项目已经有统一的 AgentLoop，也已经能在单条业务线内完成闭环，例如：

- 资料搜索 -> 歧义分析 -> ask user -> 重新搜索 -> 返回文档
- 故障码识别 -> ECU 澄清 -> 诊断
- 维修知识标题检索 -> 正文加载 -> 回答

但如果用户的问题变成下面这种复合型请求，当前架构就会开始吃力：

1. “帮我找东风天锦电路图，并结合这个故障码看一下优先排查哪里”
2. “先给我维修思路，如果有对应资料也一起列出来”
3. “这个报码是什么意思，有没有相关维修经验和对应线路图”

这类问题不是单能力问题，而是组合型问题。系统需要在一轮或多轮对话里，联合使用：

- 资料搜索能力
- 故障诊断能力
- 维修知识能力
- 参数查询能力
- ask user 澄清能力

当前项目的主要问题不是底座不支持，而是现有工程结构更偏向“单主业务线收敛”，而不是“多能力联合编排”。

## 3. 当前架构的限制

### 3.1 单入口路由过强

当前请求在很早阶段就会被判定为某一条主业务线，例如：

- 文档搜索
- 故障诊断
- 普通问答

这种设计适合快速分流，但不适合复合问题。因为一个请求一旦被归到某条主线，后续推理、工具选择、结果整形就会优先向这条主线收敛。

结果就是：

- “查资料 + 诊断” 容易被压成纯资料搜索
- “维修知识 + 参数分析” 容易被压成纯问答
- “故障码 + 资料 + ask user” 很难自然形成一条统一链路

### 3.2 工具结果缺少统一证据层

当前各类工具返回结果后，多数会比较快进入各自业务线的后处理逻辑。

这意味着：

- 资料搜索结果主要服务于“文档返回”
- 诊断结果主要服务于“诊断返回”
- 维修知识结果主要服务于“文本回答”

缺少一层统一的“证据池”来汇总：

- 找到了哪些资料
- 哪些维修知识相关
- 哪些诊断线索成立
- 哪些信息仍然缺失

没有统一证据层，就很难做“综合回答”。

### 3.3 结果终态过早

当前有些能力链路很快就会进入终态，例如：

- 直接返回 `documents`
- 直接返回 `ask_user`
- 直接返回 `message`

这对单业务链路是优点，但对多能力协同是约束。因为一个能力刚有结果，并不代表整轮回答应该结束。

### 3.4 前端按单业务生命周期设计

当前前端和后端都更偏向“当前只有一个主业务”的模型，例如当前业务是：

- 文档搜索中
- 诊断中
- 普通问答中

这对于多能力协同时不够，因为未来一轮对话可能同时处于：

- 已经拿到资料候选
- 正在等待 ECU 澄清
- 维修知识已加载

这会是一个复合状态，而不是单状态。

## 4. 目标形态

未来的目标不是简单增加更多工具，而是把当前项目演进成一个“多能力协同编排器”。

这个编排器要满足四个目标。

### 4.1 一个问题可以触发多个能力

同一条用户问题，允许系统在一个流程中同时使用多个能力。

例如：

1. 先识别故障码
2. 再拉取 ECU 候选
3. 同时查维修知识
4. 如果用户明确要求资料，再查资料
5. 最后统一组织回答

### 4.2 各能力不是直接争夺终态，而是先沉淀证据

未来的核心不应该是“哪个能力先出最终结果”，而应该是：

- 哪个能力提供了什么证据
- 哪个能力发现了什么缺失信息
- 哪个能力需要用户进一步澄清

最终由编排层决定下一步继续采集证据，还是进入最终回答。

### 4.3 澄清问题统一调度

`ask_user` 仍然保留为统一的人机交互入口，但要从“某条业务线自己的暂停点”升级为“全局编排层的暂停点”。

也就是说，未来 ask user 应该能够表达：

- 当前缺的是哪类信息
- 这个信息会被哪些能力共同使用
- 这次回答完成后应该恢复到哪个编排节点

### 4.4 最终答案是组合型答案

最终返回不应该只是一类结果，而应该能支持组合型结构，例如：

- 先给结论
- 再给维修建议
- 再给故障码/参数/ECU 线索
- 再附上相关资料
- 再说明还缺什么信息

也就是从“单类型响应”升级为“组合型回答”。

## 5. 总体演进原则

### 5.1 不推翻现有 AgentLoop

当前 AgentLoop 已经解决了很多困难问题：

- 会话连续性
- `ask_user` 暂停与恢复
- 流式输出
- 多轮恢复
- 工具调用运行时

这些不应该推翻，而应该保留并向上增加一层“编排层”。

### 5.2 单能力闭环继续保留

资料搜索、故障诊断、维修知识这些能力内部各自的逻辑不需要立即拆掉。

更合理的方式是：

- 把它们保留为能力模块
- 对外统一产出“证据”与“建议动作”
- 让编排层来决定是否继续调用其他能力

### 5.3 优先增强编排，不急着增加新能力

当前项目的短板不是能力数量少，而是能力之间配合不够。

因此优先级应该是：

1. 先让已有能力能组合
2. 再新增参数查询等能力
3. 最后做复杂图式编排

## 6. 建议的目标架构

建议把未来架构拆成六层。

| 层级 | 作用 |
| --- | --- |
| 请求接入层 | 接收请求、鉴权、建立会话上下文 |
| 能力规划层 | 判断本轮需要哪些能力，不再只输出单一业务线 |
| 能力执行层 | 执行资料搜索、诊断、维修知识、参数查询等能力 |
| 证据汇聚层 | 把各能力结果统一整理成可复用证据 |
| 编排决策层 | 决定下一步继续查、ask user、还是最终作答 |
| 回答生成层 | 将证据整合成面向用户的组合型答案 |

这六层里，真正新增的关键是三层：

- 能力规划层
- 证据汇聚层
- 编排决策层

## 7. 关键模块设计

### 7.1 能力规划器

能力规划器的职责不是直接给出最终答案，而是为当前问题生成一个“候选能力集合”。

它的输出不再是：

- 文档搜索
- 普通问答
- 故障诊断

而应该是类似：

- 主任务：故障诊断
- 辅助能力：维修知识
- 辅助能力：资料搜索
- 是否允许直接回答：是
- 是否必须澄清：否

也就是说，规划器输出的是“任务画像”，不是“单一标签”。

建议未来至少支持以下字段：

| 字段 | 含义 |
| --- | --- |
| `primary_goal` | 当前用户最核心诉求 |
| `enabled_capabilities` | 本轮允许参与的能力集合 |
| `required_inputs` | 进入关键推理前必须补齐的信息 |
| `optional_enrichment` | 可以补充但非必须的能力 |
| `response_style` | 最终更偏资料型、诊断型、综合型还是纯解释型 |

### 7.2 能力模块

建议把当前已有业务线能力抽象成统一的能力模块。

每个能力模块不再直接承担“最终回答”职责，而是统一输出三类东西：

1. `evidence`
2. `missing_info`
3. `next_action_hint`

例如：

#### 资料搜索能力

输出：

- 找到哪些文档
- 文档覆盖哪些品牌、车系、车型
- 当前是否还存在歧义
- 如果还不明确，需要用户补什么维度

#### 故障诊断能力

输出：

- 是否识别到故障码
- 是否需要 ECU
- 当前可形成哪些诊断结论
- 还需要哪些补充信息

#### 维修知识能力

输出：

- 是否找到高度相关经验
- 相关经验覆盖哪些故障现象
- 能否给出优先检查建议
- 是否仍需更多工况信息

#### 参数查询能力

输出：

- 哪些参数定义被命中
- 当前问题涉及哪些量纲、阈值、正常范围
- 这些参数是否能支撑诊断

### 7.3 统一证据池

这是未来架构里最关键的新增层。

建议为每一轮编排维护一个统一证据池。它的作用是：

- 汇总所有能力已经拿到的信息
- 标记哪些结论是强证据，哪些只是候选线索
- 标记当前还缺哪些关键输入
- 供后续能力继续复用
- 供最后的回答生成统一使用

证据池建议至少分为五类信息：

| 类别 | 内容 |
| --- | --- |
| 用户意图证据 | 用户要资料、要诊断、要维修建议、要参数说明 |
| 资料证据 | 文档结果、文档筛选条件、文档歧义状态 |
| 诊断证据 | 故障码、ECU、诊断候选、诊断结果 |
| 维修知识证据 | 标题候选、正文摘要、经验结论、来源 |
| 参数证据 | 参数定义、正常范围、单位、适用条件 |

未来任何能力执行完，都不是直接结束，而是先写入证据池。

### 7.4 编排决策器

编排决策器负责决定下一步怎么走。

建议它的决策优先级如下：

1. 当前是否已经有足够证据形成可靠回答
2. 是否缺少阻断型信息
3. 是否需要 ask user
4. 是否还有高价值能力尚未执行
5. 是否应该结束并生成最终答案

编排决策器的输出可以抽象成几种动作：

| 动作 | 含义 |
| --- | --- |
| `run_capability` | 执行某个能力 |
| `ask_user` | 发起统一澄清 |
| `synthesize_answer` | 进入答案生成 |
| `stop_with_error` | 无法继续，返回错误 |
| `stop_with_partial_answer` | 给出部分结论并提示缺失信息 |

这才是真正的“多能力编排核心”。

### 7.5 回答生成器

回答生成器不直接面对原始工具结果，而是面对统一证据池。

它负责把多来源证据整理成面向用户的最终结构。

建议未来默认支持四段式输出：

1. 结论摘要
2. 优先处理建议
3. 相关证据
4. 还需补充的信息

如果命中了资料搜索，还可以附加：

5. 相关资料

如果命中了参数查询，还可以附加：

6. 关键参数说明

这样最终回答就不再是单一类型，而是组合型答案。

## 8. 目标状态流

建议未来把状态流从“单主业务线流转”升级为“编排态流转”。

### 8.1 新状态定义

建议引入以下高层状态：

| 状态 | 含义 |
| --- | --- |
| `planning` | 分析用户诉求，决定要启用哪些能力 |
| `collecting_evidence` | 正在执行一个或多个能力采集证据 |
| `waiting_user_input` | 缺少阻断性信息，等待用户补充 |
| `synthesizing` | 证据已足够，正在整理组合型答案 |
| `completed` | 本轮已形成最终回答 |
| `partial_completed` | 已给出阶段性结论，但仍有信息缺口 |
| `failed` | 当前流程失败 |

### 8.2 单轮内部流转

建议的内部流转可以表达为：

`planning -> collecting_evidence -> decision -> collecting_evidence / waiting_user_input / synthesizing -> completed`

其中 `decision` 不一定独立对外暴露，但在系统内部必须存在。

### 8.3 跨轮恢复流转

如果进入 `waiting_user_input`，则跨轮恢复应当变成：

`waiting_user_input -> merge_user_answer -> planning_or_decision -> collecting_evidence / synthesizing`

这里的关键变化是：

恢复后不一定直接回某一条业务线，而是先回编排层，让编排层决定：

- 是继续资料搜索
- 还是继续诊断
- 还是两者都继续

## 9. ask user 的升级方案

当前 ask user 已经很好用，但未来要从“业务线局部澄清”升级成“编排级澄清”。

建议未来 ask user 增加三个概念。

### 9.1 缺失信息分组

一次 ask user 不再只为某一个能力服务，而是可以把多个能力共同需要的信息合并提问。

例如：

- 车型/车系
- ECU 型号
- 故障现象
- 工况

一次补齐，避免分能力逐个追问。

### 9.2 恢复目标

每次 ask user 都需要记录：

- 本次补充完成后优先恢复哪个编排节点
- 哪些能力可以直接消费这次补充

### 9.3 问题级优先级

当多个能力都缺信息时，需要区分：

- 阻断型缺失
- 非阻断型缺失

只对阻断型缺失发 ask user，非阻断型缺失可以边回答边提示。

## 10. 与当前 AgentLoop 的关系

### 10.1 保留的部分

以下部分建议继续保留：

- 当前统一的 AgentLoop 入口
- 当前 `ask_user` 暂停与恢复机制
- 会话历史存储
- 流式输出接口
- 资料搜索、诊断、维修知识等现有能力适配层

### 10.2 需要调整的部分

需要重点调整四个点：

1. 把单路由改成能力规划
2. 把“直接终态返回”改成“先写证据池”
3. 把 ask user 从业务线级别升级到编排级别
4. 把最终回答从单类型结果升级为组合型结果

### 10.3 不建议一开始就动的部分

不建议第一阶段就重写所有工具或重构全部前端。

更合理的路径是：

- 先增加编排层
- 再逐个把能力模块接进来
- 最后再做前端形态升级

## 11. 是否要引入 pydantic-graph

建议分阶段看。

### 11.1 第一阶段不强依赖 graph

如果当前目标是让已有能力先协同起来，第一阶段不一定要立即引入 `pydantic-graph`。

原因很简单：

- 当前 AgentLoop 已经有运行时和跨轮恢复机制
- 当前能力数量还不算特别多
- 先用现有 AgentLoop 外加“规划层 + 证据池 + 决策层”就能做出第一版多能力协同

### 11.2 第二阶段适合引入 graph

当出现下面这些情况时，就很适合引入 `pydantic-graph`：

- 编排状态越来越多
- 分支规则越来越复杂
- 需要可视化状态图
- 需要持久化图运行状态
- 需要严格控制中断与恢复点

那时可以把编排层正式升级为图式状态机。

### 11.3 建议策略

建议采用：

`先用现有 AgentLoop 做轻编排 -> 再把编排层迁移到 pydantic-graph`

这样风险最小。

## 12. 分阶段落地方案

### 阶段一：轻编排 MVP

目标：

让一个请求能同时触发“主能力 + 辅助能力”，并输出组合型回答。

本阶段建议完成：

1. 新增能力规划器
2. 新增统一证据池
3. 新增编排决策器
4. 保持现有能力模块基本不动
5. 先支持三种组合：
   - 故障诊断 + 维修知识
   - 资料搜索 + 维修知识
   - 故障诊断 + 资料搜索

本阶段完成后，系统就已经能回答：

- “报码是什么意思，优先怎么查，有没有相关资料”
- “先给维修建议，再附上相关文档”

### 阶段二：统一 ask user

目标：

把原来各业务线分散的澄清逻辑，升级成统一编排式澄清。

本阶段建议完成：

1. 定义统一缺失信息模型
2. 把多能力需要的字段一次性收集
3. 把恢复点从业务工具调用升级成编排节点恢复

本阶段完成后，用户不会被多次打断，而是尽量一次把关键缺失补齐。

### 阶段三：组合型响应协议

目标：

让前端能够原生展示组合型回答，而不是只展示单一 `message` 或 `documents`。

本阶段建议完成：

1. 增加组合型 response 类型
2. 支持答案区块化展示
3. 支持资料区块、诊断区块、参数区块、维修知识区块并存

### 阶段四：图式编排

目标：

把轻编排升级成严格状态机。

本阶段建议完成：

1. 用 `pydantic-graph` 抽象编排节点
2. 引入可视化状态图
3. 增加可持久化恢复点
4. 提升复杂多分支流程的可维护性

## 13. 推荐的第一批组合场景

为了尽快验证价值，建议先做三类高频组合场景。

### 13.1 故障码 + 维修知识

适用问题：

- “P0101 是什么问题，先查哪里”
- “这个报码一般怎么修”

协同模式：

- 先识别故障码
- 有诊断能力则走诊断
- 同时查维修知识
- 最终给出综合排查建议

### 13.2 资料搜索 + 维修知识

适用问题：

- “帮我找这车的线路图，并说下这个故障一般看哪里”

协同模式：

- 先搜资料
- 同时查维修知识
- 如果资料歧义则 ask user
- 最终同时返回资料和维修建议

### 13.3 故障码 + 资料搜索

适用问题：

- “报码 P20EE，对应后处理线路图有吗”

协同模式：

- 先识别故障码
- 再定位可能相关系统
- 再搜相关资料
- 如果需要 ECU/车型信息则统一 ask user

## 14. 对前端的影响

未来前端需要从“单业务展示”升级成“区块型结果展示”。

建议前端逐步支持：

1. 组合型答案卡片
2. 多来源证据分区
3. 一个回答里同时展示：
   - 结论
   - 维修建议
   - 相关资料
   - 诊断信息
   - 仍需补充的信息

但第一阶段不必大改前端。可以先让后端仍然返回 `message`，只是在 `metadata` 里先携带组合证据，前端逐步演进。

## 15. 对后端接口的建议

为了支持长期演进，建议逐步引入下面几个概念。

### 15.1 `plan`

本轮编排决定启用了哪些能力。

### 15.2 `evidence`

当前已经收集到的结构化证据。

### 15.3 `missing_info`

当前仍然阻断流程的信息。

### 15.4 `sections`

最终回答的区块化结果。

这四类结构可以先放在响应扩展字段里，后续稳定后再升级成正式协议。

## 16. 风险与控制

### 16.1 风险：组合后流程变长

能力一多，单轮流程会更长，流式体验和超时压力都会增大。

控制方式：

- 引入能力优先级
- 不是每次都全开
- 优先执行高价值能力

### 16.2 风险：ask user 变复杂

多能力共同澄清后，提问内容容易变得过长。

控制方式：

- 统一按“阻断信息优先”
- 尽量一次收齐
- 限制字段组数量

### 16.3 风险：回答过于冗长

组合型回答容易把所有证据都堆给用户。

控制方式：

- 回答生成器默认先出结论和优先动作
- 证据和资料做折叠式补充

## 17. 推荐结论

基于当前项目，最合理的演进路线不是重写一套新系统，而是：

1. 保留现有 AgentLoop 作为运行底座
2. 在其上增加能力规划器、证据池、编排决策器
3. 先做轻量多能力协同
4. 等状态复杂度上来后，再把编排层迁移到 `pydantic-graph`

这样做的好处是：

- 复用现有能力和状态恢复机制
- 改造风险可控
- 能快速做出第一版“组合回答”
- 后续还能自然升级成图式状态机

## 18. 一句话总结

当前项目要演进成多能力协同回答系统，关键不是继续堆工具，而是把系统从“单业务线收敛器”升级成“证据驱动的能力编排器”。

最先该做的三件事是：

- 把单一意图路由升级成能力规划
- 把各能力结果统一沉淀到证据池
- 把最终回答改成基于证据池的组合型回答
