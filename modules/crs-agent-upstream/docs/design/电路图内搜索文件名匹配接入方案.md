# 电路图内搜索文件名匹配接入方案

## 1. 接入目标

先不改变现有资料搜索。

在现有资料搜索已经返回候选文档后，用候选文档的 `filename` 去解析库 `items.name` 找到 `latest_pdf_id`，再调用电路图内搜索接口，把命中的页码、坐标和摘要挂回原搜索结果。

## 2. 字段对应

`latest_pdf_id` 不来自现有 CRS 搜索结果的 `file_id`。

现有搜索结果：

```text
file_id       CRS 本地物理文件 hash
filename      文档文件名
pic_folder_url 预览 URL
ref_file_id   外部文件 ID
```

解析库：

```text
items.name
items.latest_pdf_id
items.latest_result_path
items.latest_status
items.updated_at
```

第一阶段先用：

```text
现有搜索结果 filename -> 解析库 items.name -> latest_pdf_id
```

字段关系：

```text
result.file_id        只用于 CRS 现有文档体系，不传给图内搜索接口
result.filename       用来匹配解析库 items.name
items.latest_pdf_id   传给外部图内搜索接口的 pdf_id
body_keyword          传给外部图内搜索接口的 keyword
```

## 3. 接入位置

不要改现有文档召回逻辑。

接入点放在：

```text
AgentLoopService._execute_doc_search_workflow
  -> 文档搜索完成
  -> 澄清判断完成
  -> validity.has_valid_results = true
  -> build_documents_content 之前
```

也就是：

```text
search_data.results
  -> CircuitBodySearchEnhancer.enhance(...)
  -> search_data.results 增加 body_search
  -> DocSearchResponseAdapter.build_documents_content(...)
```

落到当前代码就是：

```text
backend/app/agent/runtime/service.py
  AgentLoopService._execute_doc_search_workflow(...)

backend/app/agent/adapters/doc_search_response_adapter.py
  DocSearchResponseAdapter.build_documents_content(...)
```

## 4. 执行链路

```text
用户问题
  ↓
现有 query planner 生成资料搜索 query
  ↓
现有资料搜索返回 results
  ↓
如果需要澄清：直接返回澄清，不做图内搜索
  ↓
如果无需澄清且结果有效：
  取 Top1-Top3
  ↓
用 result.filename 匹配解析库 items.name
  ↓
拿 latest_pdf_id
  ↓
调用外部图内搜索接口：
  pdf_id = items.latest_pdf_id
  keyword = body_keyword
  ↓
压缩命中结果
  ↓
把 body_search 挂回对应 result
```

当前 `body_keyword` 由资料搜索 query planner 显式输出：

```text
primary_query / queries: 用于资料搜索
body_keyword: 用于图内搜索
input_mode: text | text_image | image
```

图内搜索 keyword 解析仍保留回退链路：优先 `body_keyword` / `circuit_body_keyword`，再回退到 `original_query` 或 `workflow_state.query`。

## 5. 解析库文件名匹配 SQL

MVP 使用精确文件名匹配：

```sql
SELECT
  item_id,
  name,
  latest_pdf_id,
  latest_result_path,
  latest_status,
  updated_at
FROM items
WHERE name = :filename
  AND latest_status = 'completed'
  AND latest_pdf_id <> ''
  AND latest_result_path <> ''
ORDER BY updated_at DESC
LIMIT 1;
```

如果重名，取 `updated_at` 最近的一条。

## 6. 文件名归一化

匹配前做轻量归一化，但不要过度处理。

输入来源：

```text
result.filename || result.title
```

处理规则：

```text
去掉首尾空格
去掉末尾 .pdf / .PDF
保留中文、英文、数字、括号、下划线
```

第一版只做：

```python
filename = filename.strip()
filename = re.sub(r"\.(pdf|PDF)$", "", filename)
```

不要一开始做复杂模糊匹配，避免误配。

## 7. 后端模块拆分

新增模块建议：

```text
backend/app/agent/domain/circuit_body_search/
  __init__.py
  models.py
  parsed_doc_resolver.py
  search_client.py
  reducer.py
  enhancer.py
```

职责：

```text
ParsedDocResolver
  用 filename 查解析库 items，返回 latest_pdf_id

CircuitBodySearchClient
  调外部 POST /api/search

CircuitBodyHitReducer
  将大量元素命中压缩为 best_hit / page_hit_count / raw_hit_count

CircuitBodySearchEnhancer
  对 doc_search results TopN 做增强，并把 body_search 写回 result
```

## 9. 返回结构

增强后单个搜索结果增加：

```json
{
  "body_search": {
    "status": "hit",
    "match_source": "filename",
    "pdf_id": "e62d...",
    "keyword": "涡轮增压器",
    "raw_hit_count": 126,
    "page_hit_count": 59,
    "best_hit": {
      "hit_id": "p512_r1",
      "page_index": 512,
      "page_number": 513,
      "snippet": "涡轮增压器执行器 ...",
      "highlight_boxes_px": [
        [571, 500, 1151.625, 670]
      ]
    },
    "more_hits_count": 58
  }
}
```

无匹配或无命中：

```json
{
  "body_search": {
    "status": "unsupported",
    "reason": "parsed_document_not_found"
  }
}
```

外部接口失败：

```json
{
  "body_search": {
    "status": "failed",
    "reason": "external_search_failed"
  }
}
```

前端只展示 `status = hit` 的提示。其他状态不影响原文档结果。

## 10. 需要改的现有代码

### 10.1 后端搜索结果增强

文件：

```text
backend/app/agent/runtime/service.py
```

位置：

```text
_execute_doc_search_workflow
```

在 `validity.has_valid_results` 不是 `false`，且 `analysis_envelope` 没有要求澄清后，调用：

```python
body_keyword = resolve_circuit_body_keyword(
    search_data=search_data,
    fallback_query=workflow_state.query,
)
search_data["results"] = await circuit_body_search_enhancer.enhance(
    results=search_data.get("results") or [],
    body_keyword=body_keyword,
)
```

然后继续走原来的：

```python
self._build_documents_response_from_envelope(...)
```

### 10.2 前端协议透传

文件：

```text
backend/app/agent/adapters/doc_search_response_adapter.py
```

在 `build_documents_content` 中给每个 result 透传：

```python
"body_search": item.get("body_search")
```

### 10.3 前端结果卡展示

文件：

```text
frontend/user/src/App.tsx
```

在 `SearchResult` 类型和结果映射中增加：

```text
body_search
```

结果卡内如果：

```text
result.body_search.status === "hit"
```

展示：

```text
图内精确命中 · 第{page_number}页 · 点击展开
```

## 11. 下一步

1. 实现 `ParsedDocResolver`，只做文件名精确匹配。
2. 实现 `CircuitBodySearchClient`，调用已配置的外部接口。

5. `DocSearchResponseAdapter.build_documents_content` 透传 `body_search`。
6. 前端结果卡展示“图内精确命中”入口。
7. 跑一个真实 case 验证：
   - 文件名能否匹配解析库
   - latest_pdf_id 是否正确
   - 外部图内搜索是否返回命中
   - 前端是否展示增强入口

## 12. 当前不覆盖范围

文件名精确匹配链路之后，后端已补充 `url_raw_sample` 完整 PDF 预览和签名局部图预览能力。当前仍不覆盖：

- URL/normalized_url 匹配。
- 文件名模糊匹配。
- LLM rerank。
- 自有完整文档坐标级预览器。
- 新增缓存表。

局部图裁剪通过 `GET /chat/api/circuit-body-search/preview/{token}` 按需生成；完整 PDF 打开使用 `items.url_raw_sample`。
