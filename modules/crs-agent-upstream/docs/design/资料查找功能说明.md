# 资料查找功能说明

## 1. 文档目的

本文档只聚焦当前项目中的“资料查找”能力，说明它现在是怎么实现的、完整查找链路是什么、每个阶段做了什么，以及最终会输出什么结果。

这里说的“资料查找”对应当前代码里的 `DOC_SEARCH` 业务线，不包含参数查询、维修问答和故障诊断主链路。

## 2. 资料查找功能定位

资料查找的目标，是把用户输入的资料类需求转成结构化检索请求，尽可能返回可直接查看的资料候选列表；如果查询条件不够明确，则通过结构化澄清把搜索结果收敛后再返回。

当前系统把下面这类问题优先视为资料查找需求：

- 查电路图
- 查 ECU 电路图
- 查针脚定义资料
- 查维修手册
- 查接线图、原理图、资料文件
- 结合图片中的 ECU、报码、铭牌信息去找对应资料

它不是“模型随便回答一下有什么资料”，而是一个带状态恢复的专门工作流。

## 3. 当前实现落点

资料查找相关的核心实现主要分布在下面几个文件：

- 入口编排：`backend/app/agent/runtime/service.py`
- 搜索适配层：`backend/app/agent/adapters/legacy_doc_search_adapter.py`
- 结果适配层：`backend/app/agent/adapters/doc_search_response_adapter.py`
- 领域服务入口：`backend/app/agent/domain/doc_search/service.py`
- 查询规划器：`backend/app/agent/domain/doc_search/query_planner.py`
- 兼容搜索接口：`backend/app/api/search.py`

## 4. 整体实现思路

当前资料查找不是单步调用，而是一个闭环：

1. 先识别本轮请求是不是资料查找
2. 命中后进入 `doc_search` 专门工作流
3. 根据用户问题和图片证据规划搜索 query
4. 执行搜索
5. 判断结果是否有效
6. 判断是否存在歧义，需要继续澄清
7. 如果需要澄清，就返回 `ask_user`
8. 用户选择后，从保存的快照恢复搜索
9. 继续分析，直到收敛成最终结果

所以它的真实链路是：

`识别 -> 规划 -> 搜索 -> 有效性判断 -> 歧义分析 -> ask_user -> 恢复 -> 输出`

## 5. 查找入口是怎么命中的

### 5.1 统一入口

资料查找和其它业务一样，都先进入：

- `AgentLoopService.process()`
- `AgentLoopService.stream()`

### 5.2 先做意图识别

运行时会先做一次 request 级别的意图路由，得到当前请求更像哪条业务线。

在 `process()` 里，意图识别完成后，不是马上丢给主 Agent，而是先判断是否命中资料查找专门工作流：

- `_resolve_doc_search_workflow_state(...)`

### 5.3 什么时候会进入 doc_search 工作流

进入资料查找工作流主要有两种情况：

#### 情况一：新请求本身就是资料查找

当 `_suggest_requested_business(request) == "DOC_SEARCH"` 时，系统会直接按资料查找处理。

#### 情况二：这是上一轮资料澄清的恢复请求

如果前端传了：

- `ask_user_answer`
- `session_id`
- `tool_call_id`

并且能在 `deferred_state_store` 中找到对应的 `doc_search_clarify` 状态，那么当前请求不会重新开始，而是恢复到上一次资料查找的澄清节点继续执行。

## 6. 资料查找链路详细说明

## 6.1 第一步：构造工作流状态

文件：

- `backend/app/agent/runtime/service.py`

方法：

- `_resolve_doc_search_workflow_state(...)`

这一阶段会得到一个 `DocSearchWorkflowRunState`，包含：

- 原始查询 `query`
- 当前澄清轮次 `clarify_round`
- 恢复时用到的 `deferred_state`

如果当前请求带了图片证据，并且命中资料查找业务，系统还会把图片证据转成查询增强信息，拼进资料搜索 query。

## 6.2 第二步：进入 doc_search 工作流

方法：

- `_process_doc_search_workflow(...)`
- `_stream_doc_search_workflow(...)`

这里做两件事：

1. 记录 trace，标记资料查找工作流开始
2. 调用 `_execute_doc_search_workflow(...)` 真正执行搜索闭环

## 6.3 第三步：查询规划

如果不是从快照恢复，而是一次新的资料查找请求，系统会先做 query planning。

相关方法：

- `_execute_planned_doc_search(...)`
- `_plan_doc_search_queries(...)`

相关组件：

- `PydanticAIDocSearchQueryPlanner`

这一阶段的作用不是直接搜用户原句，而是尽量把问题改写成更适合命中资料库的搜索 query。

规划时会综合三类信息：

- 用户原始问题
- 图片证据摘要
- 当前会话里已经确认的 known slots

查询规划器会输出：

- `primary_query`
- `queries`
- `rationale`

也就是：

- 主查询词
- 候选补充查询词
- 为什么这么规划

当前实现还会做多 query 合并，避免只搜一次导致漏召回。

## 6.4 第四步：执行搜索

实际搜索通过：

- `LegacyDocSearchAdapter.search(...)`

进入：

- `DocSearchService.execute(...)`
- 或 `DocSearchService.execute_external(...)`

### 本地搜索路径

当没有外部 token 时，资料查找可以走本地 DB 搜索：

- `DocSearchService.execute()`
- 内部用 `search_engine_factory` 创建搜索引擎

### 外部搜索路径

当请求中带有 `app_token` 且启用了外部资料搜索时，资料查找优先走外部 ggzj 搜索：

- `DocSearchService.execute_external()`

这一路会：

1. 根据 query 生成 cache key
2. 先查 `doc_search_cache_store`
3. 缓存没有则调用外部搜索客户端
4. 通过结果适配器把外部结果转成统一结构

## 6.5 第五步：搜索后处理

搜索原始结果不会直接返回前端，而是先进入 `DocSearchService` 的后处理链路。

领域服务最终会整理出 `DocSearchExecutionResult`，里面包含：

- `query`
- `original_query`
- `results`
- `total`
- `preprocessing`
- `search_method`
- `search_time_ms`
- `requested_filters`
- `applied_filters`
- `applied_selection_payload`
- `validity`
- `summary`
- `result_summary`

这里的关键是 `validity`，它描述当前结果到底算不算“有效结果”。

## 6.6 第六步：有效性判断

资料查找不是有结果就算成功，还会做有效性校验。

典型校验包括：

- 是否命中硬约束
- 查询里的实体是否存在于库中
- 当前返回结果是否可视为有效资料

如果 `validity.has_valid_results == False`，系统不会返回 `documents`，而是进入无效结果分支，最终输出一条 `message` 响应。

这个 `message` 里会带：

- `message`
- `should_archive_previous`
- 可选的 `existence_info`

也就是说，用户会看到“没有有效资料”的结构化提示，而不是空白结果。

## 6.7 第七步：歧义分析

如果结果是有效的，系统还要继续判断是否“已经足够明确”。

相关调用：

- `LegacyDocSearchAdapter.analyze_ambiguity(...)`
- `DocSearchService.analyze_ambiguity(...)`

歧义分析分两层：

### 第一层：规则澄清

依赖：

- `clarify_service`

如果结果太多，或者品牌、车系、型号等维度存在明显冲突，就先走规则型澄清。

### 第二层：LLM 智能澄清

当结果数量仍较多时，系统还可以使用 LLM 做进一步澄清建议，生成更智能的分组选项。

最终歧义分析输出的是：

- 是否需要澄清 `need_clarify`
- 澄清维度 `facet`
- 提示问题 `question`
- 选项列表 `options`
- 前端渲染上下文 `context`

## 6.8 第八步：返回 ask_user

如果歧义分析认为还需要用户确认，系统不会立即返回资料列表，而是先构造一个 `ask_user` 响应。

构造逻辑在：

- `DocSearchResponseAdapter.build_ask_user_question(...)`

这里会把澄清结果转成统一 `AskUserQuestion`：

- `tool_call_id`
- `question`
- `input_type = SINGLE_SELECT`
- `options`
- `context`

同时还会生成对应表单定义，给前端直接渲染单选卡片。

## 6.9 第九步：保存恢复状态

在返回 `ask_user` 之前，系统会保存一个 `DeferredState`：

- `tool_name = doc_search_clarify`
- 当前原始 query
- 当前 `clarify_round`
- ask_user 原始结构
- search snapshot

其中最关键的是 `search_snapshot`，它保存了上一次搜索的结果快照：

- query
- original_query
- results
- preprocessing
- search_method
- search_time_ms
- planned_queries
- query_plan_rationale

这样用户选择后，系统可以直接基于快照恢复，而不是从零重新跑一整轮。

## 6.10 第十步：用户补充后恢复搜索

当前端把用户的选择以 `ask_user_answer` 发回来时，系统会：

1. 用 `session_id + tool_call_id` 找到保存的 deferred state
2. 用 `DocSearchResponseAdapter.resolve_selection_payload(...)` 解析用户选项对应的 `selection_payload`
3. 用 `DocSearchResponseAdapter.resolve_search_snapshot(...)` 取回上轮搜索快照
4. 调用 `search_from_snapshot(...)` 继续搜索

这里的关键不是“再搜一次”，而是“带着上轮上下文精确恢复”。

## 7. 资料查找最终输出是什么

当前资料查找最终有三种主要输出形态。

## 7.1 输出一：`documents`

这是资料查找成功收敛后的标准输出。

构造逻辑：

- `_build_documents_response_from_envelope(...)`
- `DocSearchResponseAdapter.build_documents_content(...)`

返回体形态：

- `ChatResponse.type = "documents"`
- `business = "DOC_SEARCH"`

`content` 里主要包括：

- `query`
- `total`
- `total_hits`
- `returned_count`
- `page_size`
- `results`
- `filters`
- `summary`
- `planned_queries`
- `query_plan_rationale`

其中每个 `result` 会带：

- `file_id`
- `filename`
- `physical_path`
- `file_type`
- `brand`
- `series`
- `model`
- `hierarchy_full`
- `score`
- `ref_file_id`
- `parent_id`
- `pic_folder_url`
- `ggzj_sn`
- `ggzj_data_type`
- `ggzj_file_no`
- `ggzj_file_type`

这就是前端文档卡片最终渲染的数据来源。

## 7.2 输出二：`ask_user`

这是资料查找的中间态输出，不代表失败，而代表“结果还没收敛，需要用户补一个筛选条件”。

返回体特点：

- `ChatResponse.type = "ask_user"`
- `business = "DOC_SEARCH"`
- `clarify_facet = "ask_user_question"`
- 包含 `tool_call_id`

本质上，这是资料查找工作流暂停点。

## 7.3 输出三：`message`

当结果无效、实体不存在、没有匹配资料，或者最终只适合返回说明性文本时，资料查找会输出：

- `ChatResponse.type = "message"`
- `business = "DOC_SEARCH"`

常见场景：

- 当前库中确实无相关资料
- 搜索结果分数低或命中无效
- 实体校验发现用户提到的型号在资料库不存在

## 8. 前端兼容接口的输出

除了统一聊天接口，当前项目还保留了兼容旧前端的 `/search` 接口。

文件：

- `backend/app/api/search.py`

这个接口会把资料查找结果压平成旧格式：

- `results`
- `clarify`
- `stats`
- `validity`

它仍然依赖同一套底层资料查找服务，只是输出格式更接近旧项目。

## 9. 当前资料查找链路总结

如果只用一句话概括当前实现：

当前资料查找是一个“可规划、可澄清、可恢复、可结构化输出”的专门工作流，不是简单关键字搜索。

完整链路如下：

1. 用户提问
2. Agent Loop 识别为 `DOC_SEARCH`
3. 构建 doc_search workflow state
4. 根据用户问题、图片证据和已知槽位规划 query
5. 通过适配层执行本地或外部资料搜索
6. 做结果有效性判断
7. 做歧义分析
8. 如果有歧义，返回 `ask_user`
9. 保存快照和恢复状态
10. 用户补充后，从快照恢复
11. 直到最终返回 `documents` 或 `message`

## 10. 对外描述建议

如果你要对项目成员或业务方解释现在的资料查找，可以直接用下面这段话：

“现在的资料查找不是搜一次就返回，而是一个专门闭环。系统先判断用户是不是在找资料，再结合用户问题、图片证据和已有上下文规划搜索词，然后调用本地或外部资料检索能力拿结果。拿到结果后不会立刻返回，而是先判断结果是否有效、是否还存在品牌/车系/型号等歧义。如果不够明确，就返回一个结构化澄清问题给前端；用户选完之后，后端会基于上一次保存的搜索快照继续收敛，不用从头重跑。最终结果要么是结构化资料列表 `documents`，要么是中间态 `ask_user`，要么是在没有有效资料时返回说明型 `message`。”
