# 资料搜索文件列表获取说明

## 1. 文档目的

本文档说明 benchmark 如何获取一次资料搜索请求返回的完整文件列表，以及如何从列表中提取所有文件名。

这里的“完整文件列表”指当前轮资料搜索流程得到的候选结果池，不是前端只展示的 Top-K，也不是人工标注的 gold 列表。

## 2. 数据口径

当前轮资料搜索的候选文件列表以运行时的 `search_data["results"]` 为准。

一次资料搜索请求的主链路是：

```text
用户请求
  -> doc_search workflow
  -> 外部搜索 / 多 query 搜索
  -> 合并、去重、排序
  -> search_data["results"]
  -> documents 响应或 clarify 快照
  -> benchmark prediction.results_full
```

如果是普通资料搜索结果，列表会进入最终响应：

```python
response.content["results"]
```

如果本轮触发澄清，列表会进入澄清快照：

```python
deferred_state.payload["search_snapshot"]["results"]
```

benchmark 统一输出时，完整候选列表会写入：

```python
prediction["results_full"]
```

## 3. 运行时列表来源

外部搜索的原始入口在：

```text
backend/app/agent/domain/doc_search/service.py
DocSearchService.execute_external_raw(...)
```

核心结构是：

```python
raw = {
    "query": request.query,
    "results": results,
    "preprocessing": preprocessing,
    "search_method": "ggzj_external",
    "search_time_ms": ...,
}
```

这里的 `raw["results"]` 是单次外部搜索返回并适配后的结果列表。

生产流程中可能会执行多 query 搜索。多 query 的结果会在：

```text
backend/app/agent/runtime/service.py
AgentLoopService._merge_doc_search_envelopes(...)
```

合并成：

```python
first_data["results"] = ordered_results
```

合并逻辑会按 `file_id` 去重，并按 `score`、`matched_query_confidence` 排序。合并完成后，当前轮的候选文件列表就是：

```python
search_envelope["data"]["results"]
search_data["results"]
```

## 4. documents 响应中的列表

最终 documents 响应由：

```text
backend/app/agent/adapters/doc_search_response_adapter.py
DocSearchResponseAdapter.build_documents_content(...)
```

从 `search_data["results"]` 转换得到：

```python
content = {
    "query": ...,
    "total": ...,
    "results": formatted_results,
    ...
}
```

其中每个结果项的文件名字段是：

```python
"filename": item.get("filename") or item.get("title")
```

因此，普通 documents 响应里获取所有文件名的方式是：

```python
filenames = [
    item.get("filename")
    for item in response.content.get("results", [])
    if item.get("filename")
]
```

## 5. clarify 快照中的列表

如果搜索结果需要用户澄清，当前轮完整列表会被写入 deferred state。

写入位置：

```text
backend/app/agent/runtime/service.py
AgentLoopService._execute_doc_search_workflow(...)
```

保存结构由：

```text
backend/app/agent/adapters/doc_search_response_adapter.py
DocSearchResponseAdapter.build_search_snapshot(...)
```

生成：

```python
{
    "query": ...,
    "original_query": ...,
    "results": search_data.get("results") or [],
    "preprocessing": ...,
    "planned_queries": ...,
    ...
}
```

因此，澄清场景里获取当前轮完整列表的方式是：

```python
snapshot = DocSearchResponseAdapter.resolve_search_snapshot(deferred_state)
results = snapshot.get("results", []) if snapshot else []
```

提取所有文件名：

```python
filenames = [
    item.get("filename") or item.get("title")
    for item in results
    if item.get("filename") or item.get("title")
]
```

## 6. benchmark 中的统一结果

benchmark 当前会把不同来源的搜索结果统一成 `results_full`。

转换函数在：

```text
backend/app/benchmark/doc_search.py
_extract_results(...)
_extract_results_from_documents_content(...)
```

统一后的结果项结构是：

```python
{
    "rank": index,
    "doc_name": doc_name,
    "doc_id": file_id,
    "score": score,
    "path": path,
}
```

prediction payload 中会写入：

```python
{
    "results": scored_results,
    "results_scored": scored_results,
    "results_full": all_results,
    "returned_result_count": len(scored_results),
    "full_result_count": len(all_results),
}
```

因此，外部 benchmark 如果消费本项目 benchmark 运行结果，直接读取：

```python
prediction["results_full"]
```

获取所有文件名：

```python
filenames = [
    item.get("doc_name")
    for item in prediction.get("results_full", [])
    if item.get("doc_name")
]
```

获取带排名、ID、分数、路径的完整文件列表：

```python
files = [
    {
        "rank": item.get("rank"),
        "filename": item.get("doc_name"),
        "file_id": item.get("doc_id"),
        "score": item.get("score"),
        "path": item.get("path"),
    }
    for item in prediction.get("results_full", [])
]
```

## 7. 文件名提取规则

benchmark 侧统一文件名的函数是：

```text
backend/app/benchmark/doc_search.py
_result_doc_name(...)
```

当前优先级是：

```text
doc_name -> filename -> title -> path
```

对应代码逻辑：

```python
doc_name = (
    item.get("doc_name")
    or item.get("filename")
    or item.get("title")
    or item.get("path")
    or ""
).strip()
```

所以在 benchmark 输出中，文件名使用 `doc_name`；在业务 documents 响应中，文件名使用 `filename`；在运行时原始搜索数据中，文件名使用 `filename`，没有时回退到 `title`。

## 8. 对外提供字段

如果需要把当前轮搜索到的所有文件名提供给外部 benchmark，后端输出可以直接使用下面两个字段：

```json
{
  "search_files": [
    {
      "rank": 1,
      "filename": "xxx",
      "file_id": "xxx",
      "score": 0.91,
      "path": "xxx"
    }
  ],
  "search_filenames": [
    "xxx"
  ]
}
```

字段来源：

```python
results_full = prediction.get("results_full", [])

search_files = [
    {
        "rank": item.get("rank"),
        "filename": item.get("doc_name"),
        "file_id": item.get("doc_id"),
        "score": item.get("score"),
        "path": item.get("path"),
    }
    for item in results_full
]

search_filenames = [
    item["filename"]
    for item in search_files
    if item.get("filename")
]
```

## 9. 当前已有落盘结果

benchmark run 的 `predictions.jsonl` 中，每条 case 都包含当前轮 prediction。

完整候选文件列表字段：

```python
prediction["results_full"]
```

所有文件名字段：

```python
prediction["results_full"][i]["doc_name"]
```

Excel 报告中也会生成“完整候选”sheet，字段为：

```text
Case ID / 排名 / 资料名 / 文档ID / 分数 / 路径
```

其中“资料名”对应：

```python
prediction["results_full"][i]["doc_name"]
```
